Generation Pipeline

Generation Pipeline

docsfy transforms a GitHub repository into a polished static documentation site through a four-stage sequential pipeline. Each stage builds on the output of the previous one, progressing from raw source code to finished HTML.

 ┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
 │   Stage 1    │    │   Stage 2    │    │   Stage 3    │    │   Stage 4    │
 │  Clone Repo  │───>│  AI Planner  │───>│  AI Content  │───>│    HTML      │
 │              │    │  (plan.json) │    │  Generator   │    │  Renderer    │
 └──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘
       │                    │                    │                    │
       v                    v                    v                    v
   temp directory       plan.json          cache/pages/*.md     site/*.html

The entire pipeline is triggered by a single API call:

curl -X POST http://localhost:8000/api/generate \
  -H "Content-Type: application/json" \
  -d '{"repo_url": "https://github.com/org/repo"}'

Stage 1: Clone Repository

The pipeline begins by cloning the target repository to a temporary directory on disk. This gives the AI full filesystem access to every file in the project.

How It Works

docsfy performs a shallow clone (--depth 1) to minimize disk usage and clone time. Only the latest commit on the default branch is fetched — full history is not needed for documentation generation.

git clone --depth 1 <repo_url> <temp_directory>

Repository Access

Both public and private repositories are supported:

URL Format Access Type Authentication
https://github.com/org/repo Public None required
https://github.com/org/repo Private System git credentials
git@github.com:org/repo.git Private (SSH) SSH keys on the host

Private repository access relies on the git credentials available in the container's environment. When running in Docker, SSH keys or credential helpers can be mounted into the container via volumes.

Note: The cloned repository is placed in a temporary directory that persists throughout the pipeline. It is used as the working directory for both the AI Planner (Stage 2) and AI Content Generator (Stage 3) stages, giving the AI CLI tools direct filesystem access to explore the source code.

Commit SHA Tracking

After cloning, docsfy records the current commit SHA in its SQLite database. This SHA is used later for incremental updates — when a re-generation is requested, docsfy compares the stored SHA against the latest commit to determine whether documentation needs to be refreshed.

Stage 2: AI Planner

The second stage analyzes the repository and produces a structured documentation plan. An AI CLI tool explores the codebase — reading source files, configs, tests, and READMEs — then outputs a plan.json file that defines the pages, sections, and navigation hierarchy of the final documentation site.

How It Works

The AI CLI is invoked as a subprocess with its working directory set to the cloned repository:

subprocess.run(cmd, input=prompt, capture_output=True, text=True, cwd=repo_path)

This is wrapped for async execution:

await asyncio.to_thread(subprocess.run, ...)

The prompt instructs the AI to explore the repository and produce a JSON documentation plan. The AI has full access to read any file in the repository and decides the optimal documentation structure based on what it finds.

Provider-Specific Invocation

The exact command varies by AI provider:

Provider Command
Claude claude --model <model> --dangerously-skip-permissions -p
Gemini gemini --model <model> --yolo
Cursor agent --force --model <model> --print --workspace <path>

Note: Cursor uses a --workspace flag instead of subprocess cwd to specify the repository path. This is handled internally by the uses_own_cwd flag in the provider configuration.

The provider configuration is defined as a frozen dataclass:

@dataclass(frozen=True)
class ProviderConfig:
    binary: str
    build_cmd: Callable
    uses_own_cwd: bool = False

The plan.json Output

The AI produces a plan.json file that defines the complete documentation structure. This file is the contract between the planner and all subsequent stages. It contains:

  • Pages — each page with a slug, title, and description of what it should cover
  • Sections — logical groupings of pages (e.g., "Getting Started", "API Reference")
  • Navigation hierarchy — the order and nesting of pages in the sidebar

A typical plan.json structure looks like:

{
  "project_name": "my-project",
  "sections": [
    {
      "title": "Getting Started",
      "pages": [
        {
          "slug": "introduction",
          "title": "Introduction",
          "description": "Overview of the project, its purpose, and key features"
        },
        {
          "slug": "installation",
          "title": "Installation",
          "description": "How to install and set up the project"
        }
      ]
    },
    {
      "title": "Architecture",
      "pages": [
        {
          "slug": "architecture-overview",
          "title": "Architecture Overview",
          "description": "High-level system design and component interactions"
        }
      ]
    }
  ]
}

The plan.json is saved to the project's storage directory at /data/projects/{project-name}/plan.json and serves as the source of truth for both content generation and HTML rendering.

JSON Response Parsing

AI CLI tools don't always return clean JSON. docsfy uses a multi-strategy extraction approach to reliably parse the AI output:

  1. Direct JSON parse — attempt json.loads() on the raw output
  2. Brace-matching — scan for the outermost {...} JSON object in the output
  3. Markdown code block extraction — extract JSON from ```json ... ``` fenced blocks
  4. Regex fallback — last-resort recovery using pattern matching

This layered approach ensures robust parsing regardless of whether the AI wraps its response in explanatory text, markdown formatting, or other artifacts.

Tip: If you encounter persistent JSON parsing failures, check the AI CLI's raw output. Some models add conversational text before or after the JSON. The multi-strategy parser handles most cases, but unusual formatting may require adjusting the prompt.

Stage 3: AI Content Generator

With the documentation plan in hand, the third stage generates the actual content. For every page defined in plan.json, the AI CLI is invoked with a prompt that instructs it to write comprehensive markdown documentation for that specific topic.

How It Works

Each page is generated independently:

  1. Read the page definition from plan.json (title, slug, description)
  2. Construct a prompt that references the page topic and instructs the AI to explore the codebase as needed
  3. Invoke the AI CLI with cwd set to the cloned repository
  4. Parse the markdown output
  5. Cache the result to disk

Concurrent Generation

Pages are generated concurrently using Python's async capabilities with semaphore-limited concurrency. This prevents overwhelming the AI CLI or the system while still generating multiple pages in parallel:

# Conceptual flow — pages generated concurrently with bounded parallelism
semaphore = asyncio.Semaphore(max_concurrent)

async def generate_page(page):
    async with semaphore:
        success, content = await run_ai_cli(prompt, cwd=repo_path)
        if success:
            save_to_cache(page.slug, content)

await asyncio.gather(*[generate_page(p) for p in plan.pages])

This concurrent approach significantly reduces total generation time for repositories with many documentation pages.

Cached Output

Generated markdown files are cached on disk at:

/data/projects/{project-name}/cache/pages/*.md

Each page's markdown is saved using its slug as the filename (e.g., introduction.md, api-reference.md). This cache serves two purposes:

  1. Incremental updates — unchanged pages don't need to be regenerated
  2. Debugging — the raw AI-generated markdown can be inspected before rendering

AI CLI Timeout

Content generation is subject to a configurable timeout. Since complex pages may require the AI to explore many files, the default timeout is generous:

Setting Default Description
AI_CLI_TIMEOUT 60 Timeout in minutes per AI CLI invocation

The return type from each AI invocation is a tuple[bool, str] — a success flag and the output content. Failed generations are logged and the page is skipped in the final output.

Warning: Long timeouts mean a generation request can take significant time to complete, especially for large repositories with many planned pages. Monitor the project status via GET /api/status to track progress.

Availability Check

Before starting the content generation pipeline, docsfy runs a lightweight availability check against the configured AI provider. A simple "Hi" prompt is sent to verify the CLI tool is installed, accessible, and properly authenticated. This fast-fails the pipeline with a clear error rather than waiting for a timeout on the first real prompt.

Stage 4: HTML Renderer

The final stage transforms the AI-generated markdown and the plan.json navigation structure into a polished, fully-featured static HTML documentation site.

How It Works

The renderer combines three inputs:

  1. plan.json — defines navigation structure and page ordering
  2. Cached markdown files — the content from Stage 3
  3. Jinja2 templates with bundled assets — the visual theme and interactive features

The output is a self-contained static site written to:

/data/projects/{project-name}/site/

Output Structure

site/
├── index.html              # Landing page / first page
├── introduction.html       # One HTML file per documentation page
├── installation.html
├── api-reference.html
├── ...
├── assets/
│   ├── style.css           # Complete visual theme
│   ├── search.js           # Client-side full-text search
│   ├── theme-toggle.js     # Dark/light mode switching
│   └── highlight.js        # Code syntax highlighting
└── search-index.json       # Pre-built search index

Features

The rendered HTML site includes a rich set of features out of the box:

Feature Implementation
Sidebar navigation Generated from plan.json sections and pages
Dark/light theme Client-side toggle with theme-toggle.js
Full-text search Client-side via lunr.js with search-index.json
Syntax highlighting highlight.js for fenced code blocks
Responsive design Mobile-friendly layout via CSS
Card layouts Visual cards for feature overviews
Callout boxes Note, warning, and info callouts

Templating

docsfy uses Jinja2 for HTML templating. Templates are bundled with the application and define the page shell — header, sidebar, content area, and footer. The markdown content is converted to HTML using the Python markdown library and injected into the Jinja2 page template.

Search Index

During rendering, docsfy builds a search-index.json file that indexes the content of all pages. This powers the client-side search feature — users can search across the entire documentation site without a backend server, making the static output fully self-contained.

Incremental Updates

docsfy supports incremental updates to avoid regenerating an entire documentation site when only part of the repository has changed. The process works as follows:

  1. SHA comparison — On re-generation, docsfy fetches the latest commit SHA and compares it against the stored SHA in SQLite
  2. Plan re-evaluation — If the SHA has changed, the AI Planner (Stage 2) runs again to check whether the documentation structure needs to change
  3. Selective regeneration — Only pages whose content may be affected by the changes are regenerated
  4. Cache reuse — Unchanged pages are served from the markdown cache at cache/pages/*.md
Re-generate Request
       │
       v
  Compare commit SHA
       │
  ┌────┴────┐
  │ Changed │  No ──> Return existing site
  └────┬────┘
       │ Yes
       v
  Re-run AI Planner
       │
  ┌────┴──────────┐
  │ Plan changed? │  No ──> Regenerate affected pages only
  └────┬──────────┘
       │ Yes
       v
  Regenerate all affected pages
       │
       v
  Re-render HTML site

Tip: Incremental updates significantly reduce generation time for active repositories. The first generation builds the full documentation; subsequent runs only update what has changed.

Storage Layout

The complete filesystem layout for a generated project ties all four stages together:

/data/
├── docsfy.db                              # SQLite: project metadata, status, SHA
└── projects/
    └── {project-name}/
        ├── plan.json                      # Stage 2 output: documentation structure
        ├── cache/
        │   └── pages/
        │       ├── introduction.md        # Stage 3 output: AI-generated markdown
        │       ├── installation.md
        │       └── api-reference.md
        └── site/                          # Stage 4 output: rendered HTML
            ├── index.html
            ├── introduction.html
            ├── installation.html
            ├── api-reference.html
            ├── assets/
            │   ├── style.css
            │   ├── search.js
            │   ├── theme-toggle.js
            │   └── highlight.js
            └── search-index.json

The SQLite database tracks project-level metadata:

Field Description
Project name Derived from the repository name
Repository URL The source GitHub URL
Status generating, ready, or error
Last generated Timestamp of the last successful generation
Last commit SHA Used for incremental update comparison
Generation history Logs from each generation run

Configuration

The pipeline behavior is controlled via environment variables:

# AI provider selection
AI_PROVIDER=claude          # Options: claude, gemini, cursor
AI_MODEL=claude-opus-4-6[1m]  # Model ID passed to the AI CLI
AI_CLI_TIMEOUT=60           # Timeout per AI invocation (minutes)

Provider Authentication

Each AI provider requires its own credentials:

Claude (via API key):

ANTHROPIC_API_KEY=sk-ant-...

Claude (via Vertex AI):

CLAUDE_CODE_USE_VERTEX=1
CLOUD_ML_REGION=us-east5
ANTHROPIC_VERTEX_PROJECT_ID=my-gcp-project

Gemini:

GEMINI_API_KEY=...

Cursor:

CURSOR_API_KEY=...

Warning: AI CLI tools are installed unpinned (always the latest version) inside the container. While this ensures access to the newest features, it means pipeline behavior may change between container rebuilds. Pin your Docker image tags in production for reproducible builds.

Error Handling

Each stage can fail independently, and docsfy handles failures at every level:

Stage Failure Mode Behavior
Clone Invalid URL, auth failure, network error Pipeline aborts, project status set to error
AI Planner AI CLI unavailable, timeout, unparseable JSON Pipeline aborts, project status set to error
AI Content Timeout on individual page Page skipped, other pages continue
AI Content AI CLI crash Page skipped with logged error
HTML Renderer Template error, missing pages Pipeline aborts, project status set to error

The availability check before content generation catches the most common failure — a misconfigured or unauthenticated AI CLI — before any significant work is done.
