Technology Stack

Overview

docsfy is built on a modern Python stack designed for async operation, lightweight deployment, and zero-dependency static output. The service combines a FastAPI backend with Jinja2 templating and SQLite metadata storage to orchestrate AI-driven documentation generation and serve polished static HTML sites.

Component Technology Role
Web framework FastAPI + uvicorn Async HTTP API and static file serving
Templating Jinja2 HTML page rendering from markdown content
Database SQLite (async) Project metadata and generation state
Build system hatchling Python package building
Package manager uv Dependency resolution and virtual environments
Code highlighting highlight.js Client-side syntax highlighting
Search Client-side (lunr.js or similar) In-browser full-text search
Markdown Python markdown library Markdown-to-HTML conversion

FastAPI

FastAPI serves as the core web framework, providing both the REST API for managing documentation generation and the static file serving layer for rendered documentation sites.

Why FastAPI

FastAPI was chosen for its native async support, automatic OpenAPI documentation, and high performance via uvicorn. Since documentation generation involves long-running AI CLI subprocess calls, the async architecture allows the server to remain responsive while generation tasks execute in the background via asyncio.to_thread().

API Surface

docsfy exposes seven endpoints organized into three groups:

POST /api/generate              # Start doc generation for a repo URL
GET  /api/status                # List all projects and their generation status
GET  /api/projects/{name}       # Get project details (last generated, commit SHA, pages)
DELETE /api/projects/{name}     # Remove a project and its generated docs
GET  /api/projects/{name}/download  # Download site as .tar.gz archive
GET  /docs/{project}/{path}    # Serve generated static HTML docs
GET  /health                   # Health check

The generation endpoint accepts a repository URL and triggers the four-stage pipeline (clone, plan, generate, render) as a background task. The /docs/{project}/{path} route serves the final static HTML directly from the filesystem, making docsfy both a generator and a host.

Running the Server

The application is started through uvicorn with uv managing the runtime environment:

uv run --no-sync uvicorn docsfy.main:app --host 0.0.0.0 --port 8000

Note: The --no-sync flag tells uv to skip dependency resolution at startup, relying on the pre-installed environment from the Docker build stage.

AI CLI Integration Pattern

FastAPI's async capabilities are used to run AI CLI subprocesses without blocking the event loop:

@dataclass(frozen=True)
class ProviderConfig:
    binary: str
    build_cmd: Callable
    uses_own_cwd: bool = False

AI commands are executed asynchronously:

# Prompts passed via subprocess stdin
subprocess.run(cmd, input=prompt, capture_output=True, text=True)

# Wrapped in asyncio.to_thread for non-blocking execution
await asyncio.to_thread(subprocess.run, ...)

Each provider call returns a tuple[bool, str] indicating success status and the raw output, which then goes through multi-strategy JSON extraction (direct parse, brace-matching, markdown code block extraction, regex fallback).

Jinja2

Jinja2 handles Stage 4 of the generation pipeline — converting AI-generated markdown content and the documentation plan into a polished static HTML site.

Template Rendering

Templates are bundled with the docsfy package and produce static HTML pages with:

  • Sidebar navigation generated from the plan.json hierarchy
  • Dark/light theme toggle with persistent preference
  • Responsive design for mobile and desktop viewing
  • Callout boxes for notes, warnings, and informational content
  • Card layouts for overview and index pages

The renderer reads each markdown page from cache, converts it to HTML using a Python markdown library, and passes the result through Jinja2 templates along with the navigation structure from plan.json.

Output Structure

Each rendered project produces a self-contained static site:

/data/projects/{project-name}/site/
  ├── index.html
  ├── *.html                    # One HTML file per documentation page
  ├── assets/
  │   ├── style.css             # Theme and layout styles
  │   ├── search.js             # Client-side search logic
  │   ├── theme-toggle.js       # Dark/light mode switching
  │   └── highlight.js          # Syntax highlighting
  └── search-index.json         # Pre-built search index

Tip: The generated sites are fully self-contained. You can download them via the /api/projects/{name}/download endpoint as a .tar.gz archive and host them on any static file server — no docsfy runtime required.

SQLite

SQLite provides lightweight, zero-configuration metadata storage. The database file lives at /data/docsfy.db and is accessed asynchronously to avoid blocking the FastAPI event loop.

What SQLite Stores

The database tracks project-level metadata rather than documentation content (which lives on the filesystem):

Field Purpose
Project name Unique identifier derived from the repository
Repo URL Source repository (SSH or HTTPS)
Status Current state: generating, ready, or error
Last generated Timestamp of the most recent successful generation
Last commit SHA Git SHA used for incremental update detection
Generation history Logs from past generation runs

Why SQLite

SQLite is the right fit for docsfy because:

  • No external service — the database is a single file, simplifying deployment and containerization
  • Persistent via volume mount — the /data directory is mounted as a Docker volume, so project state survives container restarts
  • Async access — libraries like aiosqlite provide async wrappers that integrate cleanly with FastAPI's event loop
  • Concurrent reads — SQLite handles concurrent read access well, which suits the read-heavy pattern of status checks and doc serving

Incremental Updates

SQLite plays a key role in the incremental update strategy:

  1. The last commit SHA is stored per project after each successful generation
  2. On re-generation, the current repo HEAD is compared against the stored SHA
  3. If unchanged, the generation is skipped entirely
  4. If changed, the AI Planner re-evaluates the documentation structure and only affected pages are regenerated

Note: Documentation content itself is stored on the filesystem under /data/projects/, not in SQLite. The database acts purely as a metadata and state coordination layer.

hatchling

hatchling is the Python build backend used to package docsfy as a distributable Python package.

Build Configuration

The build system is declared in pyproject.toml:

[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

hatchling was chosen for its simplicity and modern standards compliance. It supports PEP 621 project metadata natively, requires minimal configuration compared to setuptools, and integrates well with the uv package manager.

What hatchling Provides

  • Package building — Creates wheel and sdist distributions for deployment
  • Metadata management — Reads project name, version, dependencies, and entry points from pyproject.toml
  • Asset inclusion — Bundles Jinja2 templates, CSS, JavaScript, and other static assets with the package so they're available at runtime

Tip: hatchling's defaults are sensible for most projects. Unless you need custom build hooks or complex file inclusion rules, the two-line [build-system] declaration is all the configuration required.

uv

uv is the sole package manager for docsfy — pip is not used anywhere in the project. uv handles dependency resolution, virtual environment management, and script execution.

Role in Development

In the development workflow, uv is used for:

  • Dependency installation — uv sync resolves and installs all dependencies from pyproject.toml
  • Script execution — uv run executes commands within the managed virtual environment
  • Test execution — Tox is configured to use uv for running test suites

Role in Production

The Docker entrypoint uses uv to launch the application:

uv run --no-sync uvicorn docsfy.main:app --host 0.0.0.0 --port 8000

In the multi-stage Docker build, dependencies are installed during the build phase:

# Base image: python:3.12-slim (multi-stage build)
# uv installs dependencies during build
# Runtime uses --no-sync to skip resolution

Why uv Over pip

  • Speed — uv resolves and installs dependencies significantly faster than pip
  • Deterministic — Lock file support ensures reproducible builds across environments
  • Unified tool — Replaces pip, pip-tools, virtualenv, and parts of tox in a single binary

Warning: docsfy exclusively uses uv. Running pip install directly may produce inconsistent environments. Always use uv sync or uv run to manage dependencies.

highlight.js

highlight.js provides client-side syntax highlighting for code blocks in the generated documentation. It runs entirely in the browser with no server-side processing required.

Integration

highlight.js is bundled as a static asset within each generated documentation site at assets/highlight.js. When a documentation page loads, highlight.js automatically detects and highlights code blocks based on language annotations.

How It Works

The Jinja2 templates include highlight.js in the rendered HTML pages. During Stage 4 (HTML Rendering), the markdown-to-HTML conversion produces standard <pre><code> blocks with language classes:

<!-- Markdown input -->
```python
def hello():
    print("Hello, world!")

def hello():
    print("Hello, world!")

highlight.js then processes these blocks at page load time, applying syntax-appropriate styling that respects the current theme (dark or light mode).

### Why Client-Side Highlighting

- **Zero server cost** — Highlighting happens in the browser, keeping the rendering pipeline simple
- **Self-contained output** — The generated static sites work without any server-side runtime
- **Language auto-detection** — highlight.js can detect languages even without explicit annotation, providing a good fallback for code blocks in AI-generated content

## Client-Side Search

docsfy implements full-text search entirely in the browser, requiring no server-side search infrastructure. This keeps the generated documentation sites fully self-contained and hostable anywhere.

### Architecture

The search system has two phases:

**Build time (Stage 4 — HTML Renderer):**
- The renderer processes all markdown pages and extracts searchable text content
- A pre-built search index is written to `search-index.json` at the root of each project's site directory
- The index contains page titles, section headings, and text content mapped to their URLs

**Runtime (in the browser):**
- `search.js` loads `search-index.json` when the user initiates a search
- The search logic (using lunr.js or a similar library) indexes the content client-side
- Results are ranked by relevance and displayed inline with links to matching pages and sections

### File Layout

/data/projects/{project-name}/site/ ├── search-index.json # Pre-built index generated during rendering └── assets/ └── search.js # Client-side search logic and UI

### Search Index Format

The `search-index.json` file contains structured data for each page:

```json
[
  {
    "title": "Getting Started",
    "url": "getting-started.html",
    "body": "Full text content of the page for indexing..."
  }
]

Why Client-Side Search

Consideration Benefit
No server required Downloaded sites work offline and on any static host
No external service No Elasticsearch, Algolia, or other search infrastructure
Fast for small-to-medium sites Documentation sites typically have dozens to hundreds of pages — well within browser search capacity
Simple implementation The index is a JSON file; the search is a JavaScript library

Note: Client-side search works well for the typical documentation site size (tens to hundreds of pages). For extremely large documentation projects, the search index file size may become a consideration, but this is uncommon for single-repository documentation.

Supporting Technologies

Python Markdown

A Python markdown library converts the AI-generated markdown content into HTML during the rendering stage. This happens server-side as part of the static site generation pipeline, before Jinja2 templates wrap the content in the full page layout.

uvicorn

uvicorn is the ASGI server that runs the FastAPI application. It provides high-performance async HTTP handling and is the standard production server for FastAPI applications.

Docker

docsfy runs in a multi-stage Docker container built on python:3.12-slim:

services:
  docsfy:
    build: .
    ports:
      - "8000:8000"
    env_file: .env
    volumes:
      - ./data:/data
      - ~/.config/gcloud:/home/appuser/.config/gcloud:ro
      - ./cursor:/home/appuser/.config/cursor
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 30s
      timeout: 10s
      retries: 3

The container runs as a non-root user (appuser) with GID 0 for OpenShift compatibility. System dependencies include bash, git, curl, nodejs, npm, and ca-certificates to support repository cloning and AI CLI tool installation.
