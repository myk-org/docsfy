Incremental Updates

Incremental Updates

docsfy uses an incremental update strategy to avoid regenerating entire documentation sites when only part of a repository has changed. By tracking commit SHAs and comparing documentation plans, docsfy intelligently determines which pages need regeneration — saving time, reducing AI costs, and delivering faster updates.

How It Works

The incremental update pipeline follows a layered change-detection approach. Rather than regenerating every page on each request, docsfy compares the current repository state against the last successful generation and narrows down the work to only what has changed.

POST /api/generate (repo URL)
        │
        v
┌─────────────────────────┐
│  Fetch repo & get HEAD  │
│  commit SHA             │
└────────────┬────────────┘
             │
             v
┌─────────────────────────┐     match
│  Compare SHA against    ├────────────► No changes — skip generation
│  stored SHA in SQLite   │
└────────────┬────────────┘
             │ different
             v
┌─────────────────────────┐     unchanged
│  Re-run AI Planner      ├────────────► Regenerate only affected pages
│  Compare new plan.json  │
│  against cached plan    │
└────────────┬────────────┘
             │ changed
             v
┌─────────────────────────┐
│  Full regeneration      │
│  (new structure)        │
└─────────────────────────┘

The process breaks down into three layers of detection:

  1. Commit SHA comparison — Has the repository changed at all?
  2. Plan structure comparison — Has the documentation structure changed?
  3. Affected page identification — Which specific pages need new content?

Commit SHA Tracking

Every time docsfy generates documentation for a project, it records the HEAD commit SHA of the repository in its SQLite database. This SHA serves as the primary checkpoint for detecting changes.

Storage

The commit SHA is stored alongside other project metadata in the SQLite database at /data/docsfy.db:

Project name, repo URL, status (generating / ready / error)
Last generated timestamp, last commit SHA
Generation history and logs

When a regeneration request arrives via POST /api/generate, docsfy performs a shallow clone of the repository and compares the current HEAD SHA against the stored value:

1. Track last commit SHA per project in SQLite
2. On re-generate: fetch repo, compare current SHA against stored SHA

If the SHAs match, the repository hasn't changed since the last generation, and docsfy can skip regeneration entirely.

Querying Project State

You can inspect a project's current commit SHA and generation status through the API:

GET /api/projects/{name}

This endpoint returns project details including the last generated timestamp and the commit SHA from the most recent successful generation.

Change Detection Pipeline

When the commit SHA has changed, docsfy proceeds through its four-stage generation pipeline — but with intelligent shortcuts for incremental updates.

Stage 1: Clone Repository

docsfy performs a shallow clone to minimize bandwidth and disk usage:

Shallow clone (--depth 1) to a temporary directory

Note: Shallow cloning with --depth 1 means docsfy only fetches the latest commit. This is sufficient for documentation generation since docsfy needs the current state of the codebase, not its full history.

Stage 2: AI Planner (Structure Comparison)

This is the critical decision point for incremental updates. docsfy re-runs the AI Planner against the updated repository to produce a new plan.json:

AI CLI runs with cwd set to the cloned repo directory
AI has full access to explore the entire repository
Output: plan.json containing pages, sections, and navigation hierarchy

The new plan is compared against the previously cached plan at /data/projects/{name}/plan.json. Two outcomes are possible:

Scenario Action
Plan structure changed (pages added, removed, or reordered) Full regeneration of all pages
Plan structure unchanged (same pages, same hierarchy) Regenerate only affected pages

Stage 3: Selective Content Generation

When the plan structure is unchanged, docsfy identifies which pages are likely affected by the repository changes and regenerates only those:

If plan structure is unchanged and only specific files changed,
regenerate relevant pages only

Pages that are unaffected by the changes are served from the cached markdown files:

/data/projects/{project-name}/
  plan.json             # doc structure from AI
  cache/
    pages/*.md          # AI-generated markdown (cached for incremental updates)

For each page defined in plan.json, docsfy either: - Regenerates the page if its content may be affected by the repository changes - Reuses the cached markdown from /data/projects/{name}/cache/pages/*.md

Tip: Pages are generated concurrently using async execution with semaphore-limited concurrency. During incremental updates, only the subset of affected pages consumes AI resources, making the process significantly faster.

Stage 4: HTML Rendering

After content generation (whether full or partial), the HTML renderer rebuilds the complete static site:

/data/projects/{project-name}/
  site/                 # final rendered HTML
    index.html
    *.html
    assets/
      style.css
      search.js
      theme-toggle.js
      highlight.js
    search-index.json

The renderer uses Jinja2 templates to convert all markdown pages (both newly generated and cached) along with plan.json into the final static HTML site with sidebar navigation, search index, and themed styling.

Note: The HTML rendering stage always runs in full, even during incremental updates. This ensures the search index, navigation, and cross-page links remain consistent across the entire site.

Cache Structure

docsfy maintains a layered cache on the filesystem that enables incremental updates:

/data/projects/{project-name}/
  plan.json             # doc structure from AI
  cache/
    pages/*.md          # AI-generated markdown (cached for incremental updates)
  site/                 # final rendered HTML
    index.html
    *.html
    assets/
      style.css
      search.js
      theme-toggle.js
      highlight.js
    search-index.json
Path Purpose Used During Incremental Updates
plan.json Documentation structure and navigation hierarchy Compared against new plan to detect structural changes
cache/pages/*.md AI-generated markdown per page Reused for pages unaffected by changes
site/ Final rendered HTML output Rebuilt entirely after content generation

AI Provider Integration

Incremental updates work identically across all supported AI providers. The AI Planner and Content Generator stages use the configured provider:

# AI Configuration
AI_PROVIDER=claude
AI_MODEL=claude-opus-4-6[1m]
AI_CLI_TIMEOUT=60
Provider Binary CWD Handling
Claude claude subprocess cwd = repo path
Gemini gemini subprocess cwd = repo path
Cursor agent --workspace flag

The provider configuration follows a consistent pattern:

@dataclass(frozen=True)
class ProviderConfig:
    binary: str
    build_cmd: Callable
    uses_own_cwd: bool = False

During incremental updates, the AI CLI is invoked only for the planner step and for pages that need regeneration — unaffected pages skip AI invocation entirely.

Tip: Incremental updates can substantially reduce AI CLI usage. If a repository change only affects one module, only the pages documenting that module are regenerated, while all other pages are served from cache.

Triggering Incremental Updates

Incremental updates are triggered automatically when you call the generate endpoint for a project that has already been generated:

POST /api/generate

The flow is:

  1. First generation — Full pipeline runs: clone, plan, generate all pages, render HTML. The commit SHA and all artifacts are stored.
  2. Subsequent generations — Incremental pipeline: clone, compare SHA, compare plan, regenerate only affected pages, re-render HTML. The stored SHA is updated.

Warning: If the cached artifacts under /data/projects/{name}/ are deleted or corrupted, docsfy will fall back to a full regeneration regardless of the commit SHA comparison.

Monitoring Generation Status

You can monitor whether a project is undergoing a full or incremental generation through the status API:

GET /api/status          # List all projects and their generation status
GET /api/projects/{name} # Get project details (last generated, commit SHA, pages)

Project status values:

Status Description
generating Generation in progress (full or incremental)
ready Documentation is up to date and being served
error Generation failed — check logs for details

Deployment Considerations

Persistent Storage

For incremental updates to work, the data volume must persist between container restarts:

services:
  docsfy:
    build: .
    ports:
      - "8000:8000"
    env_file: .env
    volumes:
      - ./data:/data

The ./data:/data volume mount ensures that both the SQLite database (with commit SHAs) and the filesystem cache (with generated markdown and HTML) survive container restarts.

Warning: Without persistent storage, docsfy loses all cached artifacts and commit SHA records, forcing a full regeneration on every request. Always mount the /data volume in production deployments.

Health Checks

The health check endpoint verifies that the service is operational, including access to the SQLite database where commit SHAs are tracked:

healthcheck:
  test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
  interval: 30s
  timeout: 10s
  retries: 3

Summary

The incremental update strategy minimizes unnecessary work at every level:

Layer Check Skip Condition
Repository Commit SHA comparison SHA unchanged — no regeneration needed
Structure Plan comparison Plan unchanged — skip full regeneration
Content Affected page detection Page unaffected — reuse cached markdown
Rendering Always runs N/A — ensures site consistency

This layered approach means that minor repository changes (such as fixing a typo in a single source file) result in only the relevant documentation page being regenerated, while the rest of the site is assembled from cache — delivering fast, cost-efficient updates.
