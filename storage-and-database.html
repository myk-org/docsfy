Storage & Database

Storage & Database

docsfy uses a two-layer storage architecture: a SQLite database for project metadata and tracking, and a filesystem directory tree for cached markdown content, rendered HTML sites, and search indexes. Both live under a single /data mount point for simple container deployment.

Storage Overview

/data/
  docsfy.db                          # SQLite database (project metadata)
  projects/                          # Filesystem storage (generated content)
    {project-name}/
      plan.json                      # Documentation structure from AI Planner
      cache/
        pages/*.md                   # AI-generated markdown (cached)
      site/                          # Final rendered static HTML site
        index.html
        *.html
        assets/
          style.css
          search.js
          theme-toggle.js
          highlight.js
        search-index.json            # Client-side search index

All storage is rooted at /data, which is mapped as a Docker volume:

# docker-compose.yaml
services:
  docsfy:
    volumes:
      - ./data:/data

Tip: Since all persistent state lives under /data, backing up this single directory captures everything — the database, cached content, and rendered sites.

SQLite Database

Location

The database file lives at /data/docsfy.db. docsfy accesses it asynchronously via aiosqlite, which is well-suited for the FastAPI async request lifecycle.

Schema

The database stores project metadata only — it does not store page content or rendered HTML. The tracked fields are:

Field Description
Project name Unique identifier derived from the repository
Repository URL Source GitHub repository (SSH or HTTPS)
Status Current generation state: generating, ready, or error
Last generated Timestamp of the most recent successful generation
Last commit SHA HEAD commit SHA from the last generation run
Generation history Logs from previous generation runs

Project Status Lifecycle

A project transitions through these statuses during generation:

POST /api/generate
       │
       v
  ┌──────────┐     success     ┌───────┐
  │generating │ ──────────────> │ ready │
  └──────────┘                  └───────┘
       │
       │ failure
       v
  ┌───────┐
  │ error │
  └───────┘
  • generating — Pipeline is actively running (clone, plan, content generation, render).
  • ready — Site is built and available for serving at GET /docs/{project}/.
  • error — Generation failed; logs are stored for debugging.

Incremental Update Tracking

The database plays a central role in docsfy's incremental update strategy. When a project is re-generated:

  1. The stored last commit SHA is compared against the repository's current HEAD.
  2. If the SHA has changed, the AI Planner re-runs to check if the documentation structure changed.
  3. Only pages whose source content may be affected are regenerated.
  4. If the plan structure is unchanged and only specific files changed, only the relevant pages are rebuilt.

Note: The commit SHA comparison happens before any AI invocation, making the "nothing changed" case fast and free of AI costs.

Filesystem Layout

Per-Project Directory

Each project gets its own directory at /data/projects/{project-name}/ containing three distinct areas:

/data/projects/{project-name}/
  plan.json
  cache/
    pages/*.md
  site/
    ...

plan.json — Documentation Structure

The plan.json file is the output of Stage 2 (AI Planner) and defines the entire documentation structure: pages, sections, and navigation hierarchy.

This file is: - Generated by the AI CLI after analyzing the full repository - Used by the content generator to know which pages to write - Used by the HTML renderer to build sidebar navigation - Persisted for incremental update comparisons (to detect if the doc structure changed)

cache/pages/ — Markdown Cache

cache/
  pages/
    getting-started.md
    api-reference.md
    configuration.md
    ...

The cache/pages/ directory holds the AI-generated markdown files produced in Stage 3 (AI Content Generator). Each page defined in plan.json gets a corresponding .md file.

These cached markdown files serve two purposes:

  1. Input for rendering — The HTML renderer converts these into the final static site.
  2. Incremental update cache — On re-generation, unchanged pages can be skipped if their source material hasn't changed, avoiding unnecessary AI invocations.

Warning: Manually editing files in cache/pages/ is not recommended. They will be overwritten on the next generation run. To customize output, adjust AI prompts or post-process the rendered site instead.

site/ — Rendered HTML Output

site/
  index.html
  getting-started.html
  api-reference.html
  configuration.html
  assets/
    style.css
    search.js
    theme-toggle.js
    highlight.js
  search-index.json

The site/ directory contains the final static HTML output from Stage 4 (HTML Renderer). This is a fully self-contained static site that can be:

  • Served directly by docsfy at GET /docs/{project}/{path}
  • Downloaded as a .tar.gz archive via GET /api/projects/{name}/download for self-hosting

The renderer uses Jinja2 templates with bundled CSS and JavaScript assets to produce a polished documentation site with:

  • Sidebar navigation (generated from plan.json)
  • Dark/light theme toggle
  • Client-side search
  • Code syntax highlighting (via highlight.js)
  • Responsive design
  • Callout boxes (note, warning, info)
  • Card layouts

Search Index

How It Works

docsfy uses client-side search powered by lunr.js (or similar). The search index is generated at build time during the HTML rendering stage and served as a static JSON file.

search-index.json

The search index lives at:

/data/projects/{project-name}/site/search-index.json

This JSON file is built from the markdown content of all pages during Stage 4. It contains pre-processed, tokenized content that the client-side search library can query without any server-side infrastructure.

search.js

The corresponding search JavaScript is bundled as a static asset:

/data/projects/{project-name}/site/assets/search.js

This script loads search-index.json on the client side and provides instant search results as the user types — no server roundtrips required.

Tip: Because search is entirely client-side, downloaded/self-hosted sites get full search functionality with no additional setup.

Docker Volume Mapping

The recommended deployment maps /data as a bind mount or named volume:

services:
  docsfy:
    build: .
    ports:
      - "8000:8000"
    env_file: .env
    volumes:
      - ./data:/data
      - ~/.config/gcloud:/home/appuser/.config/gcloud:ro
      - ./cursor:/home/appuser/.config/cursor
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 30s
      timeout: 10s
      retries: 3

The container runs as a non-root user (appuser) for OpenShift compatibility. The /data directory must be writable by this user.

Warning: If you use a bind mount (./data:/data), ensure the host directory exists and has appropriate permissions before starting the container.

Storage Management

Removing a Project

Use the API to cleanly remove a project and all its associated storage:

DELETE /api/projects/{name}

This removes: - The project record from the SQLite database - The entire /data/projects/{name}/ directory tree (plan, cache, and site)

Disk Space Considerations

Each generated project consumes disk space across three areas:

Area Contents Relative Size
plan.json Documentation structure Small (a few KB)
cache/pages/ AI-generated markdown Moderate (depends on page count)
site/ Rendered HTML + assets + search index Largest (HTML + bundled JS/CSS)

The SQLite database (docsfy.db) itself remains small, as it stores only metadata — not page content.

Tip: To reclaim space without losing the ability to re-render, you can delete the site/ directory for a project. The cached markdown and plan can be used to regenerate the HTML without re-invoking the AI.
