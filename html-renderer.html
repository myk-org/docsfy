HTML Renderer

HTML Renderer

The HTML Renderer is Stage 4 of the docsfy generation pipeline. It transforms AI-generated Markdown pages and the plan.json navigation structure into a polished, self-contained static HTML documentation site.

Overview

The renderer takes two inputs:

  1. plan.json — the documentation structure produced by the AI Planner (Stage 2), defining pages, sections, and navigation hierarchy
  2. Markdown files — per-page content generated by the AI Content Generator (Stage 3), cached at /data/projects/{name}/cache/pages/*.md

It produces a complete static site at /data/projects/{name}/site/ that can be served directly by the docsfy API or downloaded as a .tar.gz archive for self-hosting.

/data/projects/{project-name}/site/
  index.html
  *.html
  assets/
    style.css
    search.js
    theme-toggle.js
    highlight.js
  search-index.json

Jinja2 Templating

The renderer uses Jinja2 as its templating engine to convert Markdown content into fully-structured HTML pages. Each page defined in plan.json is rendered through a base template that provides consistent layout, navigation, and asset loading.

Template Context

When rendering a page, the following context variables are passed to the Jinja2 template:

Variable Type Description
page_title str Title of the current page
page_content str Rendered HTML from Markdown source
project_name str Name of the documentation project
navigation list[dict] Sidebar navigation tree from plan.json
current_page str Slug/path of the active page (for highlighting)
pages list[dict] All pages metadata for search index generation

Markdown Processing

Markdown content is converted to HTML using the Python markdown library before being injected into the template. The renderer processes several extended Markdown features:

  • Fenced code blocks — converted to <pre><code> elements with language classes for syntax highlighting
  • Callout boxes — blockquote patterns for note, warning, and info callouts are transformed into styled <div> elements
  • Card layouts — structured content blocks rendered as visual cards
  • Tables — GitHub-flavored Markdown tables with responsive wrappers

Callout Syntax

The renderer recognizes three callout types in Markdown source:

> **Note:** This provides additional context or helpful information.

> **Warning:** This highlights potential issues or breaking changes.

> **Info:** This offers supplementary details about the topic.

These are rendered as styled callout boxes with distinct colors and icons for each type.

Example: Rendered Page Structure

<!DOCTYPE html>
<html lang="en" data-theme="light">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>{{ page_title }} - {{ project_name }}</title>
    <link rel="stylesheet" href="assets/style.css">
    <link rel="stylesheet" href="assets/highlight.css">
</head>
<body>
    <nav class="sidebar">
        <!-- Sidebar navigation -->
    </nav>
    <main class="content">
        <header class="top-bar">
            <!-- Search bar and theme toggle -->
        </header>
        <article class="page-content">
            {{ page_content }}
        </article>
    </main>
    <script src="assets/highlight.js"></script>
    <script src="assets/search.js"></script>
    <script src="assets/theme-toggle.js"></script>
</body>
</html>

Bundled CSS/JS Assets

The renderer bundles all frontend assets into the assets/ directory of the generated site. This ensures the documentation is fully self-contained with no external CDN dependencies — the entire site works offline.

Asset Files

File Purpose
style.css Complete site styling including layout, typography, responsive design, dark/light themes
search.js Client-side full-text search engine and UI
theme-toggle.js Dark/light theme switching with localStorage persistence
highlight.js Code syntax highlighting via highlight.js
search-index.json Pre-built search index generated at render time

Assets are copied from the docsfy package's bundled templates into each project's site directory during rendering, ensuring every generated site is independent.

Sidebar Navigation

The sidebar provides hierarchical navigation generated from the plan.json structure. It supports nested sections and highlights the currently active page.

Navigation Structure from plan.json

The plan.json file produced by the AI Planner defines the navigation hierarchy:

{
  "project_name": "my-project",
  "pages": [
    {
      "slug": "introduction",
      "title": "Introduction",
      "description": "Project overview and getting started",
      "section": "Getting Started"
    },
    {
      "slug": "installation",
      "title": "Installation",
      "description": "How to install and configure",
      "section": "Getting Started"
    },
    {
      "slug": "api-reference",
      "title": "API Reference",
      "description": "Complete API documentation",
      "section": "Reference"
    }
  ]
}

Sidebar Rendering

Pages are grouped by their section field into collapsible navigation groups. The renderer:

  1. Parses plan.json to extract the page list
  2. Groups pages by section to create a two-level hierarchy
  3. Renders each group as a collapsible section with its pages listed beneath
  4. Marks the current page with an active class for visual highlighting
<nav class="sidebar">
    <div class="sidebar-header">
        <h2>{{ project_name }}</h2>
    </div>
    <ul class="nav-tree">
        {% for section, pages in navigation %}
        <li class="nav-section">
            <span class="section-title">{{ section }}</span>
            <ul>
                {% for page in pages %}
                <li class="{% if page.slug == current_page %}active{% endif %}">
                    <a href="{{ page.slug }}.html">{{ page.title }}</a>
                </li>
                {% endfor %}
            </ul>
        </li>
        {% endfor %}
    </ul>
</nav>

Mobile Sidebar

On smaller viewports, the sidebar collapses into a hamburger menu overlay. A toggle button appears in the top bar to open and close the sidebar. See Responsive Design for breakpoint details.

Dark/Light Theme Toggle

The renderer includes a theme toggle that allows users to switch between dark and light modes. Theme preference is persisted in the browser's localStorage.

How It Works

The theme system uses a data-theme attribute on the <html> element:

<html lang="en" data-theme="light">

The theme-toggle.js script manages the toggle behavior:

  1. On page load — checks localStorage for a saved preference, falls back to the system preference via prefers-color-scheme, and defaults to light
  2. On toggle click — switches the data-theme attribute between light and dark and saves the choice to localStorage
  3. Across pages — the saved preference is applied immediately on every page load to prevent flash of wrong theme (FOWT)

CSS Theme Variables

The theme is implemented using CSS custom properties that change based on the data-theme attribute:

:root,
[data-theme="light"] {
    --bg-primary: #ffffff;
    --bg-secondary: #f8f9fa;
    --bg-sidebar: #f1f3f5;
    --text-primary: #212529;
    --text-secondary: #495057;
    --border-color: #dee2e6;
    --link-color: #228be6;
    --code-bg: #f1f3f5;
}

[data-theme="dark"] {
    --bg-primary: #1a1b1e;
    --bg-secondary: #25262b;
    --bg-sidebar: #18191c;
    --text-primary: #c1c2c5;
    --text-secondary: #909296;
    --border-color: #373a40;
    --link-color: #4dabf7;
    --code-bg: #2c2e33;
}

All site styles reference these variables rather than hard-coded color values, ensuring seamless theme switching without page reloads.

Toggle Button

The theme toggle renders as a sun/moon icon button in the top bar:

<button class="theme-toggle" aria-label="Toggle dark mode">
    <span class="icon-sun">☀️</span>
    <span class="icon-moon">🌙</span>
</button>

Tip: The toggle respects the user's operating system preference on first visit. If a user has their OS set to dark mode, the documentation site will load in dark mode automatically.

Client-Side Search

The renderer generates a pre-built search index at render time and bundles a client-side search engine. This enables instant full-text search without any server-side component — ideal for static hosting.

Search Index Generation

During rendering, the renderer builds search-index.json by extracting searchable content from each page:

search_entries = []
for page in plan["pages"]:
    search_entries.append({
        "slug": page["slug"],
        "title": page["title"],
        "section": page["section"],
        "body": strip_html_tags(rendered_content),
    })

The resulting search-index.json contains the full text of every page, stripped of HTML tags, alongside metadata for display in search results.

Search UI

The search interface consists of a search input in the top bar and a results dropdown:

<div class="search-container">
    <input type="text" class="search-input"
           placeholder="Search documentation..."
           aria-label="Search documentation">
    <div class="search-results" hidden>
        <!-- Results populated by search.js -->
    </div>
</div>

How Search Works

The search.js script provides client-side full-text search:

  1. Index loading — on first search interaction, search-index.json is fetched and parsed
  2. Query matching — user input is matched against page titles, section names, and body text
  3. Result ranking — matches are ranked by relevance (title matches rank higher than body matches)
  4. Result display — matching pages are shown in a dropdown with title, section, and a text snippet with the matching term highlighted

Note: The search index is loaded lazily on first interaction to avoid impacting initial page load performance.

Keyboard Navigation

The search results support keyboard navigation:

  • Ctrl+K or / — focus the search input
  • ↑ / ↓ — navigate through results
  • Enter — navigate to the selected result
  • Escape — close the results dropdown

Code Syntax Highlighting

The renderer uses highlight.js for automatic syntax highlighting of fenced code blocks. Highlighting runs client-side after page load.

Supported Languages

highlight.js provides syntax highlighting for a wide range of languages out of the box, including Python, JavaScript, TypeScript, Go, Rust, Java, C/C++, Shell/Bash, JSON, YAML, HTML, CSS, SQL, and many more.

Markdown Code Blocks

Fenced code blocks in Markdown source are converted to properly-classed HTML elements:

```python
from docsfy.renderer import render_site

render_site(project_name="my-project", plan=plan, pages=pages)
```

This renders as:

<pre><code class="language-python">from docsfy.renderer import render_site

render_site(project_name="my-project", plan=plan, pages=pages)
</code></pre>

The highlight.js script then detects the language-* class and applies syntax highlighting automatically on page load:

document.addEventListener("DOMContentLoaded", function () {
    document.querySelectorAll("pre code").forEach(function (block) {
        hljs.highlightElement(block);
    });
});

Theme-Aware Highlighting

Code block colors adapt to the current theme. The highlight.js theme CSS uses the same CSS custom properties as the rest of the site:

pre {
    background: var(--code-bg);
    border: 1px solid var(--border-color);
    border-radius: 6px;
    padding: 1rem;
    overflow-x: auto;
}

code {
    font-family: 'SF Mono', 'Fira Code', 'Cascadia Code', monospace;
    font-size: 0.875rem;
    line-height: 1.5;
}

Tip: If a code block has no language specified, highlight.js will attempt automatic language detection. For best results, always specify the language in your fenced code blocks.

Responsive Design

The generated documentation sites are fully responsive, adapting to desktops, tablets, and mobile devices using CSS media queries and a flexible layout system.

Layout Structure

The site uses a two-column layout with a fixed sidebar and scrollable main content area:

body {
    display: flex;
    min-height: 100vh;
}

.sidebar {
    width: 280px;
    position: fixed;
    height: 100vh;
    overflow-y: auto;
}

.content {
    margin-left: 280px;
    flex: 1;
    max-width: 900px;
    padding: 2rem;
}

Breakpoints

The renderer uses the following responsive breakpoints:

Breakpoint Width Behavior
Desktop ≥ 1024px Full two-column layout with visible sidebar
Tablet 768px – 1023px Narrower sidebar, adjusted content margins
Mobile < 768px Sidebar hidden behind hamburger menu, full-width content

Mobile Adaptations

On mobile viewports (< 768px), the following adaptations are applied:

@media (max-width: 768px) {
    .sidebar {
        transform: translateX(-100%);
        position: fixed;
        z-index: 1000;
        transition: transform 0.3s ease;
    }

    .sidebar.open {
        transform: translateX(0);
    }

    .content {
        margin-left: 0;
        padding: 1rem;
    }

    .hamburger-btn {
        display: block;
    }

    pre {
        font-size: 0.8rem;
    }

    table {
        display: block;
        overflow-x: auto;
    }
}

Key mobile behaviors:

  • Sidebar collapses into an off-canvas overlay toggled by a hamburger button
  • Code blocks use smaller font sizes and allow horizontal scrolling
  • Tables become horizontally scrollable within their container
  • Content padding is reduced for more readable line lengths
  • Search expands to full width

Typography

The renderer applies a clean, readable typographic scale:

body {
    font-family: -apple-system, BlinkMacSystemFont, 'Segoe UI', Roboto,
                 Oxygen, Ubuntu, Cantarell, sans-serif;
    font-size: 16px;
    line-height: 1.7;
    color: var(--text-primary);
}

h1 { font-size: 2rem; margin-top: 2rem; }
h2 { font-size: 1.5rem; margin-top: 1.75rem; }
h3 { font-size: 1.25rem; margin-top: 1.5rem; }

Output Structure

After rendering completes, the full static site is written to /data/projects/{name}/site/. The output directory contains everything needed to host the documentation independently:

site/
├── index.html              # Landing page / first page in plan
├── introduction.html       # Individual page per plan entry
├── installation.html
├── api-reference.html
├── assets/
│   ├── style.css           # Complete site styles (dark + light themes)
│   ├── search.js           # Client-side search engine
│   ├── theme-toggle.js     # Theme switching logic
│   └── highlight.js        # Code syntax highlighting
└── search-index.json       # Pre-built search index

Serving the Output

The generated site can be served in two ways:

  1. Via the docsfy API — access at GET /docs/{project}/{path} directly from the running docsfy service
  2. Self-hosted — download the site as a .tar.gz archive via GET /api/projects/{name}/download and serve it from any static file host (Nginx, Caddy, S3, GitHub Pages, etc.)

Note: Because all assets are bundled and there are no external dependencies, the generated site works entirely offline once downloaded. No CDN or internet connection is required.

Incremental Re-rendering

When a project is regenerated incrementally (see Incremental Updates), the renderer only re-renders pages whose Markdown source has changed. The search index is always fully rebuilt to ensure consistency. Unchanged pages retain their existing HTML output.
