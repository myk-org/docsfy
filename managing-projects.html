Managing Projects

Managing Projects

Documentation projects in docsfy go through a defined lifecycle — from initial generation to serving and eventual cleanup. The API provides endpoints for listing all projects, inspecting individual project details, tracking generation status, and removing projects you no longer need.

Project Lifecycle

Every project passes through a series of statuses as it moves from generation to completion:

Status Description
generating The AI pipeline is actively cloning the repository, planning documentation structure, generating content, and rendering HTML.
ready Generation completed successfully. Documentation is available for browsing and download.
error Generation failed at some stage. Check the generation history for details. 
POST /api/generate          GET /api/status
      │                          │
      v                          v
 ┌────────────┐           ┌─────────────┐
 │ generating │──success──▶│    ready    │
 │            │            └─────────────┘
 │            │──failure──▶┌─────────────┐
 └────────────┘            │    error    │
                           └─────────────┘

A project that has reached ready or error status can be re-generated by posting the same repository URL to /api/generate again. docsfy uses incremental updates to regenerate only the pages affected by repository changes.

Listing Projects

Retrieve all projects and their current generation status with GET /api/status.

Request

curl http://localhost:8000/api/status

Response

The response contains an array of all tracked projects with their current status, repository URL, and last generation timestamp:

[
  {
    "name": "my-project",
    "repo_url": "https://github.com/owner/my-project",
    "status": "ready",
    "last_generated": "2026-03-04T14:30:00Z",
    "last_commit_sha": "a1b2c3d4e5f6..."
  },
  {
    "name": "another-project",
    "repo_url": "https://github.com/owner/another-project",
    "status": "generating",
    "last_generated": null,
    "last_commit_sha": null
  }
]

Tip: Poll this endpoint to monitor the progress of in-flight generation jobs. Once a project transitions from generating to ready, its documentation is available at /docs/{project}/.

Inspecting a Project

Get detailed information about a specific project with GET /api/projects/{name}.

Request

curl http://localhost:8000/api/projects/my-project

Response

The response includes full project metadata: the documentation plan, page inventory, generation timestamps, and the commit SHA that was used for generation:

{
  "name": "my-project",
  "repo_url": "https://github.com/owner/my-project",
  "status": "ready",
  "last_generated": "2026-03-04T14:30:00Z",
  "last_commit_sha": "a1b2c3d4e5f6...",
  "pages": [
    "index",
    "getting-started",
    "configuration",
    "api-reference"
  ],
  "generation_history": [
    {
      "timestamp": "2026-03-04T14:30:00Z",
      "commit_sha": "a1b2c3d4e5f6...",
      "status": "ready",
      "pages_generated": 4
    },
    {
      "timestamp": "2026-03-03T10:15:00Z",
      "commit_sha": "f6e5d4c3b2a1...",
      "status": "ready",
      "pages_generated": 4
    }
  ]
}

Project Storage Layout

Each project's artifacts are stored under /data/projects/{project-name}/:

/data/projects/{project-name}/
  plan.json             # Documentation structure from AI planner
  cache/
    pages/*.md          # AI-generated markdown (cached for incremental updates)
  site/                 # Final rendered HTML
    index.html
    *.html
    assets/
      style.css
      search.js
      theme-toggle.js
      highlight.js
    search-index.json

The plan.json file contains the navigation hierarchy and page definitions produced by the AI planner stage. Cached markdown files in cache/pages/ are preserved between generations to support incremental updates — only pages affected by repository changes are regenerated.

Generation History and Status Tracking

docsfy persists project metadata in a SQLite database at /data/docsfy.db. This includes:

  • Project name and repository URL
  • Current status (generating, ready, or error)
  • Last generated timestamp — when the most recent generation completed
  • Last commit SHA — the repository commit used for the most recent generation
  • Generation history — a log of all past generation runs with their outcomes

Tracking Incremental Updates

When you re-generate documentation for an existing project, docsfy performs an incremental update rather than a full rebuild:

  1. The repository is fetched and the current commit SHA is compared against the stored SHA
  2. If the commit has changed, the AI Planner re-analyzes the repository structure
  3. If the documentation plan structure is unchanged, only pages affected by modified source files are regenerated
  4. If the plan structure has changed (new pages, removed sections), the renderer updates navigation across all pages

This means re-generating documentation for a project with minor changes is significantly faster than the initial generation.

# Re-generate docs after pushing changes to the repository
curl -X POST http://localhost:8000/api/generate \
  -H "Content-Type: application/json" \
  -d '{"repo_url": "https://github.com/owner/my-project"}'

Note: The project name is derived from the repository URL. Posting the same URL triggers an incremental update of the existing project rather than creating a duplicate.

Deleting a Project

Remove a project and all its generated documentation with DELETE /api/projects/{name}.

Request

curl -X DELETE http://localhost:8000/api/projects/my-project

What Gets Deleted

Deleting a project removes:

  • The project record from the SQLite database (metadata, status, generation history)
  • The entire project directory under /data/projects/{project-name}/, including:
  • The documentation plan (plan.json)
  • All cached markdown files (cache/pages/*.md)
  • The rendered HTML site (site/)

Warning: This operation is irreversible. All generated documentation and cached content for the project will be permanently removed. You will need to run a full generation again to recreate the documentation.

Verifying Deletion

After deleting a project, confirm it has been removed by listing all projects:

curl http://localhost:8000/api/status

The deleted project should no longer appear in the response.

Downloading Project Documentation

Before deleting a project — or for self-hosting purposes — you can download the complete generated site as a .tar.gz archive.

Request

curl -o my-project-docs.tar.gz \
  http://localhost:8000/api/projects/my-project/download

The archive contains the full contents of the site/ directory: all HTML pages, CSS, JavaScript assets, and the search index. Extract and serve it with any static file server.

# Extract and serve locally
tar -xzf my-project-docs.tar.gz -C ./my-docs
cd my-docs && python -m http.server 3000

Tip: Download a project archive before deleting it if you want to preserve a copy of the documentation for offline use or self-hosting.

Browsing Generated Documentation

Generated documentation is served directly by docsfy at /docs/{project}/{path}:

# Open the project's documentation index
curl http://localhost:8000/docs/my-project/

# Access a specific page
curl http://localhost:8000/docs/my-project/getting-started.html

In a browser, navigate to http://localhost:8000/docs/my-project/ to browse the full documentation site with sidebar navigation, dark/light theme toggle, client-side search, and syntax-highlighted code blocks.

Common Workflows

Generate and Monitor

# Start generation
curl -X POST http://localhost:8000/api/generate \
  -H "Content-Type: application/json" \
  -d '{"repo_url": "https://github.com/owner/my-project"}'

# Poll until status is "ready" or "error"
curl http://localhost:8000/api/projects/my-project

# Browse the docs
open http://localhost:8000/docs/my-project/

Export and Clean Up

# Download the docs
curl -o docs.tar.gz http://localhost:8000/api/projects/my-project/download

# Delete the project from docsfy
curl -X DELETE http://localhost:8000/api/projects/my-project

Update After Repository Changes

# Push changes to your repository, then re-generate
curl -X POST http://localhost:8000/api/generate \
  -H "Content-Type: application/json" \
  -d '{"repo_url": "https://github.com/owner/my-project"}'

# Check that the new commit SHA is reflected
curl http://localhost:8000/api/projects/my-project

Note: Incremental updates compare the current repository HEAD against the last generated commit SHA. If no new commits have been pushed, the generation will complete quickly with no changes.
