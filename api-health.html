GET /health

Health check endpoint for container orchestration and monitoring.

Overview

The /health endpoint provides a lightweight health check for the docsfy service. It is designed for use by container orchestration platforms (Docker, Kubernetes, OpenShift) and external monitoring systems to verify that the FastAPI server is running and accepting requests.

Request

GET /health

No authentication, query parameters, or request body required.

Response

Status Code: 200 OK

{"status": "ok"}

The endpoint returns a minimal JSON response confirming the service is operational. By design, this is a shallow health check — it verifies that the FastAPI application and uvicorn server are responsive without probing backend dependencies like the SQLite database or filesystem storage.

Note: This endpoint does not validate connectivity to /data/docsfy.db or the /data/projects/ filesystem. It confirms only that the HTTP server is accepting connections on port 8000.

Usage Examples

curl

curl -f http://localhost:8000/health

Python (requests)

import requests

response = requests.get("http://localhost:8000/health")
response.raise_for_status()
print(response.json())  # {"status": "ok"}

Python (httpx, async)

import httpx

async with httpx.AsyncClient() as client:
    response = await client.get("http://localhost:8000/health")
    response.raise_for_status()
    print(response.json())  # {"status": "ok"}

Docker Health Check

The docker-compose.yaml configures automatic health checking against this endpoint:

services:
  docsfy:
    build: .
    ports:
      - "8000:8000"
    env_file: .env
    volumes:
      - ./data:/data
      - ~/.config/gcloud:/home/appuser/.config/gcloud:ro
      - ./cursor:/home/appuser/.config/cursor
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 30s
      timeout: 10s
      retries: 3
Parameter Value Description
test curl -f http://localhost:8000/health Checks the health endpoint; -f makes curl return a non-zero exit code on HTTP errors
interval 30s Time between health check probes
timeout 10s Maximum time to wait for a response before marking the check as failed
retries 3 Number of consecutive failures before the container is marked unhealthy

Tip: The curl binary is included in the container image (python:3.12-slim base with curl installed as a system dependency) specifically to support this health check.

Container States

Docker reports three health states based on the /health endpoint:

State Meaning
starting Container started but the first health check has not yet run
healthy The last 1+ health checks returned 200 OK
unhealthy 3 consecutive health checks failed (non-200 response or timeout)

Check the current health state with:

docker inspect --format='{{.State.Health.Status}}' docsfy-docsfy-1

Or via docker ps, which displays health status in the STATUS column:

docker ps
# CONTAINER ID  IMAGE   ...  STATUS                   PORTS
# a1b2c3d4e5f6  docsfy  ...  Up 2 min (healthy)       0.0.0.0:8000->8000/tcp

Kubernetes / OpenShift

For Kubernetes-based deployments, map the health endpoint to liveness and readiness probes:

apiVersion: apps/v1
kind: Deployment
metadata:
  name: docsfy
spec:
  template:
    spec:
      containers:
        - name: docsfy
          image: docsfy:latest
          ports:
            - containerPort: 8000
          livenessProbe:
            httpGet:
              path: /health
              port: 8000
            initialDelaySeconds: 5
            periodSeconds: 30
            timeoutSeconds: 10
            failureThreshold: 3
          readinessProbe:
            httpGet:
              path: /health
              port: 8000
            initialDelaySeconds: 3
            periodSeconds: 10
            timeoutSeconds: 5
            failureThreshold: 1

Note: The docsfy container runs as a non-root user (appuser) with GID 0, making it compatible with OpenShift's restricted security context constraints.

Monitoring Integration

Use the health endpoint with external monitoring tools to alert on service downtime:

# Nagios / Icinga check
check_http -H localhost -p 8000 -u /health -e 200

# Prometheus Blackbox Exporter target
- targets:
  - http://docsfy:8000/health

Troubleshooting

Symptom Possible Cause Resolution
Connection refused uvicorn not started or crashed Check container logs: docker logs docsfy-docsfy-1
Timeout (no response) Server overloaded by active generation tasks Check CPU/memory usage; generation tasks run via asyncio.to_thread and are CPU-bound
Container marked unhealthy 3 consecutive failures (see above causes) Inspect with docker inspect and review logs for errors during startup
curl: (7) Failed to connect in healthcheck Port mismatch or server binding to wrong interface Verify entrypoint uses --host 0.0.0.0 --port 8000

Warning: If the container is repeatedly cycling between healthy and unhealthy, review resource limits. AI CLI subprocess execution (Claude Code, Cursor Agent, Gemini CLI) during documentation generation is resource-intensive and may temporarily starve the event loop, causing health check timeouts.
