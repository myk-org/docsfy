AI CLI Integration

AI CLI Integration

docsfy delegates all intelligence — repository analysis, documentation planning, and content generation — to external AI CLI tools running as subprocesses. This page explains the provider abstraction layer, how commands are built and executed, how AI responses are parsed, and how timeouts protect the pipeline from hung processes.

Overview

Rather than embedding AI SDKs or calling HTTP APIs directly, docsfy shells out to locally-installed AI CLI binaries. This approach offers several advantages:

  • No SDK dependencies — the AI tools manage their own authentication, model routing, and API versioning
  • Provider portability — switching from Claude to Gemini or Cursor requires changing a single environment variable
  • Full repository context — each CLI tool runs with cwd set to the cloned repository, giving it native file exploration capabilities

The integration layer follows the same provider pattern used in jenkins-job-insight, keeping the abstraction consistent across projects in the ecosystem.

The ProviderConfig Pattern

At the core of the integration is a frozen dataclass that encapsulates everything docsfy needs to know about a given AI CLI tool:

@dataclass(frozen=True)
class ProviderConfig:
    binary: str
    build_cmd: Callable
    uses_own_cwd: bool = False

Field Reference

Field Type Description
binary str Name of the CLI executable (e.g., "claude", "gemini", "agent")
build_cmd Callable Function that constructs the full command-line argument list given a model name and optional workspace path
uses_own_cwd bool Whether the provider handles working directory via its own flag (e.g., --workspace) rather than subprocess cwd. Defaults to False

Note: The dataclass uses frozen=True, making provider configurations immutable after construction. This prevents accidental mutation during concurrent page generation where multiple tasks share the same config.

Provider Definitions

Each supported provider has a corresponding ProviderConfig instance. The build_cmd callable encapsulates the provider-specific CLI flags:

# Claude Code
ProviderConfig(
    binary="claude",
    build_cmd=lambda model, _: [
        "claude", "--model", model, "--dangerously-skip-permissions", "-p"
    ],
)

# Gemini CLI
ProviderConfig(
    binary="gemini",
    build_cmd=lambda model, _: [
        "gemini", "--model", model, "--yolo"
    ],
)

# Cursor Agent
ProviderConfig(
    binary="agent",
    build_cmd=lambda model, workspace: [
        "agent", "--force", "--model", model, "--print", "--workspace", workspace
    ],
    uses_own_cwd=True,
)

Provider Command Summary

Provider Binary Command CWD Handling
Claude claude claude --model <model> --dangerously-skip-permissions -p subprocess cwd = repo path
Gemini gemini gemini --model <model> --yolo subprocess cwd = repo path
Cursor agent agent --force --model <model> --print --workspace <path> --workspace flag, uses_own_cwd=True

Warning: The --dangerously-skip-permissions flag (Claude) and --yolo flag (Gemini) disable interactive permission prompts. These flags are required because docsfy runs AI tools non-interactively in a subprocess where no user is present to approve file reads. Only use docsfy in trusted environments, such as containers with cloned repositories.

Working Directory Strategies

docsfy uses two strategies for giving the AI tool access to the repository, determined by the uses_own_cwd flag:

Strategy 1: Subprocess cwd (default)

For Claude and Gemini, the subprocess itself is launched with cwd pointing to the cloned repository. The AI tool inherits this working directory and can explore files relative to it.

subprocess.run(cmd, cwd=repo_path, input=prompt, capture_output=True, text=True)

Strategy 2: Provider-managed workspace (uses_own_cwd=True)

For Cursor Agent, the workspace path is passed explicitly via the --workspace flag. The subprocess cwd is not set to the repo path since the tool manages its own working directory.

# Cursor builds the workspace into its command
cmd = ["agent", "--force", "--model", model, "--print", "--workspace", repo_path]
subprocess.run(cmd, input=prompt, capture_output=True, text=True)

Subprocess Execution

Core Invocation

All AI CLI calls flow through a single execution function that wraps subprocess.run with async support:

subprocess.run(cmd, input=prompt, capture_output=True, text=True)

Key aspects of the invocation:

  • Prompt delivery via stdin — the input=prompt parameter pipes the prompt text to the CLI tool's standard input, avoiding shell escaping issues with complex prompts
  • Output capture — capture_output=True captures both stdout (the AI response) and stderr (diagnostic output)
  • Text mode — text=True ensures string I/O rather than bytes

Async Wrapping

Since subprocess.run is a blocking call, docsfy wraps it with asyncio.to_thread to prevent blocking the FastAPI event loop:

result = await asyncio.to_thread(subprocess.run, cmd, input=prompt, capture_output=True, text=True)

This is critical during Stage 3 (content generation), where multiple pages are generated concurrently using async tasks with semaphore-limited concurrency.

Return Type

The execution function returns a tuple[bool, str]:

tuple[bool, str]  # (success, output)
  • success — True if the subprocess exited with return code 0
  • output — the stdout content from the AI tool (the generated text or JSON)

Tip: Check stderr in addition to the return code when debugging failed invocations. AI CLI tools often print diagnostic information, authentication errors, or rate-limit messages to stderr.

Availability Check

Before starting a full generation pipeline, docsfy performs a lightweight availability check by sending a simple "Hi" prompt to the configured AI CLI tool:

# Verify the AI CLI is installed and authenticated
success, output = await run_ai_cli(provider_config, model, prompt="Hi", repo_path="/tmp")
if not success:
    raise RuntimeError(f"AI CLI '{provider_config.binary}' is not available")

This catches common issues early: - CLI binary not installed or not on $PATH - Missing or expired authentication credentials - Model not available for the configured provider

JSON Response Parsing

AI CLI tools return free-form text, but docsfy needs structured JSON from certain stages — most critically, the plan.json output from Stage 2 (AI Planner). Since LLMs don't always produce clean JSON, docsfy uses a multi-strategy extraction approach with four fallback levels.

Strategy 1: Direct JSON Parse

The simplest case — attempt to parse the entire output as valid JSON:

import json

try:
    result = json.loads(output)
except json.JSONDecodeError:
    # Fall through to next strategy
    pass

This succeeds when the AI returns nothing but a well-formed JSON object with no surrounding text.

Strategy 2: Brace-Matching Extraction

When the AI includes explanatory text before or after the JSON, brace-matching locates the outermost {...} object:

# Find the first '{' and match it to its closing '}'
start = output.index('{')
# Walk forward, tracking brace depth, to find the matching close brace

This handles responses like:

Here's the documentation plan:

{"pages": [...], "navigation": [...]}

Let me know if you'd like any changes.

Strategy 3: Markdown Code Block Extraction

LLMs frequently wrap JSON in markdown code fences. This strategy strips the fencing:

```json
{
  "pages": [
    {"title": "Getting Started", "slug": "getting-started"}
  ]
}
```

The parser extracts content between the opening ```json (or ```) and the closing ``` markers, then parses the inner text as JSON.

Strategy 4: Regex Recovery

As a last resort, regex patterns attempt to extract and reconstruct JSON fragments from heavily malformed output. This handles edge cases like:

  • Truncated JSON (missing closing braces)
  • Multiple JSON fragments in the output
  • JSON interleaved with commentary

Note: The parsing strategies are applied in order. As soon as one succeeds, the result is returned. This layered approach means clean AI output is parsed instantly (Strategy 1), while progressively messier output is handled by later strategies.

When Parsing Is Used

Pipeline Stage Expected Output Parsing Required
Stage 2: AI Planner plan.json structure Yes — must extract structured JSON
Stage 3: AI Content Generator Markdown text No — raw text output used directly
Availability check Any text response No — only checks success/failure

Timeout Handling

Long-running AI CLI processes are a real risk — a complex repository might take significant time to analyze, or a tool might hang indefinitely. docsfy uses a configurable timeout to guard against this.

Configuration

The timeout is controlled via the AI_CLI_TIMEOUT environment variable:

# In .env or environment
AI_CLI_TIMEOUT=60  # timeout in minutes
Setting Default Unit Description
AI_CLI_TIMEOUT 60 minutes Maximum time allowed for each AI CLI subprocess call

Tip: For large repositories with many files, consider increasing the timeout beyond the default 60 minutes. Conversely, for small projects or during development, a shorter timeout (e.g., 10-15 minutes) can help catch issues faster.

Timeout Scope

The timeout applies independently to each AI CLI subprocess invocation, not to the overall pipeline:

Pipeline execution:
├── Stage 2: AI Planner ................ ← AI_CLI_TIMEOUT applies
└── Stage 3: AI Content Generator
    ├── Page 1 ......................... ← AI_CLI_TIMEOUT applies
    ├── Page 2 ......................... ← AI_CLI_TIMEOUT applies
    └── Page N ......................... ← AI_CLI_TIMEOUT applies

This means: - A single page generation can take up to the full timeout duration - Total pipeline time can exceed AI_CLI_TIMEOUT when generating multiple pages - Each concurrent page generation task has its own independent timeout

Warning: With the default 60-minute timeout and many pages generating concurrently, the total generation time for a large project could be substantial. Monitor generation status via the GET /api/status endpoint.

Default Configuration

The complete set of AI CLI integration defaults:

Setting Default Value Environment Variable

Timeout 60 minutes AI_CLI_TIMEOUT

Provider Selection at Runtime

The active provider is resolved from the AI_PROVIDER environment variable and looked up in a registry of ProviderConfig instances. This happens once at the start of each generation pipeline:

POST /api/generate
    │
    ▼
Read AI_PROVIDER env var ──► Look up ProviderConfig
    │
    ▼
Availability check (send "Hi" prompt)
    │
    ▼
Run pipeline with selected provider

Note: Changing AI_PROVIDER or AI_MODEL takes effect on the next generation request. In-progress generations continue using the provider that was active when they started.

Concurrency Model

During Stage 3, multiple pages are generated concurrently to reduce total pipeline time. The concurrency model works as follows:

# Semaphore limits how many AI CLI subprocesses run simultaneously
semaphore = asyncio.Semaphore(max_concurrent)

async def generate_page(page_info):
    async with semaphore:
        return await asyncio.to_thread(
            subprocess.run, cmd, input=prompt,
            capture_output=True, text=True
        )

# Launch all page generation tasks concurrently
tasks = [generate_page(page) for page in plan["pages"]]
results = await asyncio.gather(*tasks)

The semaphore prevents overloading the system by limiting the number of simultaneous AI CLI subprocesses. Each subprocess still has its own independent timeout via AI_CLI_TIMEOUT.
