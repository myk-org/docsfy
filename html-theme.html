HTML Theme & Features

HTML Theme & Features

Overview

Every documentation site generated by docsfy ships as a fully self-contained static HTML package. The HTML renderer (Stage 4 of the generation pipeline) converts AI-generated markdown pages and the plan.json navigation structure into a polished, production-ready site using Jinja2 templates with bundled CSS and JavaScript assets.

The rendered site is stored at /data/projects/{project-name}/site/ with the following structure:

/data/projects/{project-name}/site/
├── index.html
├── *.html                    # one HTML file per documentation page
├── assets/
│   ├── style.css             # main stylesheet (dark/light themes)
│   ├── search.js             # lunr.js client-side search
│   ├── theme-toggle.js       # dark/light mode switching
│   └── highlight.js          # code syntax highlighting
└── search-index.json         # pre-built search index

Because every asset is bundled into the site directory, generated documentation works entirely offline — no CDN dependencies or external network requests required.

Sidebar Navigation

The sidebar navigation is automatically generated from the plan.json file produced by the AI Planner in Stage 2 of the pipeline. The planner analyzes your repository and outputs a structured documentation plan containing pages, sections, and a navigation hierarchy.

How It Works

  1. The AI Planner explores your repository and produces plan.json with a hierarchical page structure
  2. The HTML Renderer reads plan.json and builds the sidebar from the defined sections and pages
  3. Each page in the hierarchy becomes a navigable link in the sidebar

Navigation Structure

The plan.json file defines the complete navigation tree:

{
  "pages": [
    {
      "title": "Getting Started",
      "section": "Introduction",
      "filename": "getting-started"
    },
    {
      "title": "Configuration",
      "section": "Usage",
      "filename": "configuration"
    }
  ]
}

The sidebar groups pages by section and preserves the ordering defined in plan.json. The current page is visually highlighted, and sections can be expanded or collapsed for easier navigation on sites with many pages.

Tip: The AI Planner determines the optimal navigation structure by analyzing your repository's code, README, existing documentation, and project structure. You don't need to manually define the navigation — it's fully automated.

Dark / Light Theme Toggle

Generated sites include a built-in theme toggle that lets readers switch between dark and light modes. The toggle is powered by theme-toggle.js, one of the bundled JavaScript assets.

How It Works

The theme system uses CSS custom properties (variables) defined in style.css to control colors across the entire site. The theme-toggle.js script:

  1. Checks for a saved theme preference in the browser's localStorage
  2. Falls back to the operating system preference via prefers-color-scheme
  3. Applies the selected theme by toggling a CSS class on the document root
  4. Persists the user's choice for future visits

CSS Custom Properties

The theme is implemented using two sets of CSS custom properties — one for light mode and one for dark mode:

:root {
  --bg-primary: #ffffff;
  --bg-secondary: #f8f9fa;
  --text-primary: #1a1a2e;
  --text-secondary: #4a4a6a;
  --border-color: #e2e8f0;
  --accent-color: #6366f1;
  --code-bg: #f1f5f9;
}

[data-theme="dark"] {
  --bg-primary: #0f172a;
  --bg-secondary: #1e293b;
  --text-primary: #e2e8f0;
  --text-secondary: #94a3b8;
  --border-color: #334155;
  --accent-color: #818cf8;
  --code-bg: #1e293b;
}

The toggle button is placed in the site header for easy access and uses an icon that reflects the current theme state (sun for light mode, moon for dark mode).

Note: The theme preference is stored per-browser using localStorage. Each generated docsfy site maintains its own independent theme setting.

Client-Side Search with lunr.js

Every generated site includes full-text search powered by lunr.js. Search runs entirely in the browser — no server-side component needed — making it work even when the site is downloaded and hosted statically.

Search Index Generation

During Stage 4 (HTML Rendering), the renderer builds a search-index.json file from all generated pages. This pre-built index contains the tokenized, stemmed, and weighted content of every page in the documentation.

/data/projects/{project-name}/site/
├── assets/
│   └── search.js             # lunr.js integration and UI
└── search-index.json         # pre-built search index

How It Works

  1. At build time, the renderer extracts text from all markdown pages and creates a lunr.js-compatible index stored as search-index.json
  2. When a user opens the site, search.js loads the pre-built index into memory
  3. As the user types a query, lunr.js performs client-side full-text search against the index
  4. Results are ranked by relevance and displayed with page titles, section context, and matching excerpts

Search Features

  • Full-text search across all documentation pages
  • Relevance ranking using lunr.js scoring algorithms
  • Instant results — no network requests after initial page load
  • Excerpt highlighting — matching terms are highlighted in search results
  • Keyboard navigation — accessible search result browsing

Tip: Because the search index is a static JSON file, search works perfectly in fully offline or air-gapped environments after the site has been downloaded via the /api/projects/{name}/download endpoint.

Code Syntax Highlighting

Code blocks in generated documentation are syntax-highlighted using highlight.js. The highlighting is applied to all fenced code blocks in the AI-generated markdown content.

Supported Languages

highlight.js provides syntax highlighting for all major programming languages. The AI Content Generator (Stage 3) produces markdown with language-annotated code fences, which the HTML Renderer converts to highlighted HTML:

```python
@dataclass(frozen=True)
class ProviderConfig:
    binary: str
    build_cmd: Callable
    uses_own_cwd: bool = False
```

This renders with full Python syntax highlighting — keywords, decorators, type hints, and string literals are all color-coded for readability.

Theme Integration

Code highlighting colors adapt to the active theme (dark or light). The highlight.js theme is coordinated with the site's CSS custom properties so code blocks blend naturally with the rest of the page in both modes.

assets/
├── style.css          # includes code block base styles
└── highlight.js       # highlight.js runtime + language grammars

Note: The highlight.js bundle is included directly in the site assets. No external CDN or network request is required for syntax highlighting to work.

Card Layouts

Card layouts provide a visually structured way to present grouped content such as feature overviews, API summaries, or quick-start guides. Cards are rendered as bordered, rounded containers arranged in a responsive grid.

Usage in Markdown

The AI Content Generator can produce card-style content in the markdown output. The HTML Renderer recognizes card markup patterns and converts them into styled card components.

Cards are useful for:

  • Feature grids — showcasing project capabilities at a glance
  • API endpoint summaries — grouping related endpoints with descriptions
  • Quick-start options — presenting multiple getting-started paths
  • Comparison tables — side-by-side feature or technology comparisons

Responsive Behavior

Card grids adapt to the viewport width:

Viewport Layout
Desktop (≥1024px) 3-column grid
Tablet (≥768px) 2-column grid
Mobile (<768px) Single-column stack

Cards maintain consistent spacing and alignment across breakpoints, ensuring readability on all screen sizes.

Callout Boxes

Callout boxes draw attention to important information within documentation pages. docsfy supports three callout types, each with distinct visual styling to communicate severity and intent.

Callout Types

Type Purpose Color
Note General supplementary information Blue
Warning Potential pitfalls or breaking changes Amber/Yellow
Info Helpful context or background details Green

Markdown Syntax

The AI Content Generator produces callout blocks using blockquote-based syntax in the generated markdown. The HTML Renderer converts these into styled callout components:

> **Note:** This feature requires Python 3.12 or later.

> **Warning:** Changing the AI provider after initial generation
> will regenerate all pages from scratch.

> **Info:** The search index is rebuilt automatically whenever
> documentation is regenerated.

Rendered Output

Each callout type renders with:

  • A colored left border indicating the callout type
  • A bold label (Note, Warning, or Info) with an associated icon
  • A tinted background that adapts to the active theme (dark/light)
  • Full support for inline markdown (links, code, bold, italic) within the callout body

Tip: Callout boxes are theme-aware. In dark mode, callout backgrounds and border colors adjust to maintain readability and contrast without appearing harsh.

Responsive Design

Generated documentation sites are fully responsive and work across desktop, tablet, and mobile devices. The responsive behavior is implemented entirely in CSS via media queries in style.css.

Breakpoints

The layout adapts at standard breakpoints:

Breakpoint Behavior
≥1024px (Desktop) Sidebar visible alongside content; full-width card grids
≥768px (Tablet) Sidebar collapsible; 2-column card grids
<768px (Mobile) Sidebar hidden behind hamburger menu; single-column layout

Sidebar Behavior

  • Desktop: The sidebar is always visible in a fixed left panel, with the main content area occupying the remaining width
  • Tablet: The sidebar can be toggled open/closed to maximize reading space
  • Mobile: The sidebar is fully hidden and accessible via a hamburger menu icon in the header

Content Adaptations

  • Code blocks gain horizontal scrolling on narrow viewports to prevent layout breakage
  • Tables become horizontally scrollable within their containers
  • Images scale to fit the viewport with max-width: 100%
  • Card grids reflow from multi-column to single-column layouts
  • Navigation collapses into a mobile-friendly overlay menu
  • Search expands to full-width on mobile for easier input

Typography

Font sizes and spacing scale relative to the viewport to maintain readability:

  • Body text uses a base size optimized for reading on each device class
  • Headings scale proportionally to maintain visual hierarchy
  • Line lengths are capped to keep text within comfortable reading widths (~65-80 characters per line)

Static Site Portability

All theme features work without any server — the generated site is fully static and self-contained. You can:

  1. Serve directly from docsfy via GET /docs/{project}/{path}
  2. Download and self-host via GET /api/projects/{name}/download (returns a .tar.gz archive)
  3. Deploy anywhere — GitHub Pages, Netlify, S3, nginx, or any static file server
# Download and extract a generated site
curl -O http://localhost:8000/api/projects/my-project/download
tar xzf my-project.tar.gz
# Serve locally with any static server
python -m http.server -d my-project/site 8080

Note: Because all assets (CSS, JavaScript, search index, highlight.js) are bundled in the assets/ directory, the site has zero external dependencies. Dark/light theme, search, and syntax highlighting all work offline.
