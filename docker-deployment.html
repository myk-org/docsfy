Docker Deployment

Docker Deployment

docsfy is designed to run as a containerized service. This page covers the Dockerfile structure, docker-compose configuration, volume management, health checks, and production deployment best practices.

Dockerfile Structure

docsfy uses a multi-stage build based on python:3.12-slim to keep the final image lean while including all necessary tooling.

Base Image and System Dependencies

The container starts from python:3.12-slim and installs the system-level packages required for repository cloning, AI CLI tools, and health checks:

Dependency Purpose
bash Shell for AI CLI installers
git Cloning target repositories
curl Health checks and CLI installer downloads
nodejs, npm Gemini CLI runtime
ca-certificates TLS for HTTPS git remotes and API calls

Package Management with uv

docsfy uses uv as its Python package manager — not pip. The entrypoint invokes uv run --no-sync to execute the application without re-resolving dependencies at startup:

ENTRYPOINT ["uv", "run", "--no-sync", "uvicorn", "docsfy.main:app", "--host", "0.0.0.0", "--port", "8000"]

This starts the FastAPI application via uvicorn, listening on all interfaces at port 8000.

AI CLI Installation

All three supported AI CLIs are installed at build time. They are unpinned to always pull the latest version:

# Claude Code
RUN curl -fsSL https://claude.ai/install.sh | bash

# Cursor Agent
RUN curl -fsSL https://cursor.com/install | bash

# Gemini CLI
RUN npm install -g @google/gemini-cli

Note: Because the CLI versions are unpinned, rebuilding the image at different times may produce containers with different CLI versions. Pin versions in production if reproducibility is critical.

Non-Root User

The container runs as a non-root user appuser with OpenShift-compatible permissions (GID 0):

RUN useradd -m -g 0 appuser
USER appuser

This ensures compatibility with OpenShift's restricted security context constraints, which require containers to run with an arbitrary UID within group 0.

Warning: If you encounter permission errors with volume mounts, ensure the mounted directories are readable/writable by GID 0 on the host.

docker-compose.yaml

The recommended way to run docsfy is with Docker Compose. The following configuration is defined in the project's docker-compose.yaml:

services:
  docsfy:
    build: .
    ports:
      - "8000:8000"
    env_file: .env
    volumes:
      - ./data:/data
      - ~/.config/gcloud:/home/appuser/.config/gcloud:ro
      - ./cursor:/home/appuser/.config/cursor
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 30s
      timeout: 10s
      retries: 3

Starting the Service

# Build and start
docker compose up --build -d

# View logs
docker compose logs -f docsfy

# Stop the service
docker compose down

Port Configuration

The service exposes port 8000 by default. To change the host-side port:

ports:
  - "3000:8000"   # Access via http://localhost:3000

Tip: If running behind a reverse proxy (e.g., nginx or Traefik), you can remove the ports mapping entirely and connect via the Docker network instead.

Environment Variables

Configuration is managed through a .env file referenced by env_file: .env in the compose file. Create it from the example template:

cp .env.example .env

AI Provider Configuration

# AI Configuration
AI_PROVIDER=claude                    # Options: claude, cursor, gemini
AI_MODEL=claude-opus-4-6[1m]         # Model selection
AI_CLI_TIMEOUT=60                    # Timeout in minutes

Authentication Variables

Only the variables for your chosen provider need to be set:

Claude (API Key):

ANTHROPIC_API_KEY=sk-ant-...

Claude (Vertex AI):

CLAUDE_CODE_USE_VERTEX=1
CLOUD_ML_REGION=us-central1
ANTHROPIC_VERTEX_PROJECT_ID=my-gcp-project

Gemini:

GEMINI_API_KEY=...

Cursor:

CURSOR_API_KEY=...

Application Settings

LOG_LEVEL=INFO                        # Options: DEBUG, INFO, WARNING, ERROR

Warning: Never commit your .env file to version control. It contains API keys and secrets. Ensure .env is listed in your .gitignore.

Volume Mounts

docsfy uses three volume mounts, each serving a distinct purpose:

Data Volume (./data:/data)

The primary persistent storage volume. Contains the SQLite database and all generated documentation:

/data/
├── docsfy.db                          # SQLite metadata database
└── projects/
    └── {project-name}/
        ├── plan.json                  # Documentation structure from AI
        ├── cache/
        │   └── pages/*.md             # AI-generated markdown (cached)
        └── site/                      # Final rendered HTML
            ├── index.html
            ├── *.html
            ├── assets/
            │   ├── style.css
            │   ├── search.js
            │   ├── theme-toggle.js
            │   └── highlight.js
            └── search-index.json

This volume is read-write and must persist across container restarts. Without it, all generated documentation and project metadata are lost.

Tip: Back up the ./data directory regularly in production. The SQLite database at /data/docsfy.db tracks project metadata including generation status, commit SHAs, and history.

Google Cloud Credentials (~/.config/gcloud:/home/appuser/.config/gcloud:ro)

Mounted read-only (:ro). Required only when using Gemini as the AI provider or Claude via Vertex AI. This bind-mounts your host's gcloud configuration into the container so the AI CLIs can authenticate with Google Cloud.

If you are not using Google Cloud services, you can safely remove this mount:

volumes:
  - ./data:/data
  # - ~/.config/gcloud:/home/appuser/.config/gcloud:ro  # Not needed
  - ./cursor:/home/appuser/.config/cursor

Cursor Configuration (./cursor:/home/appuser/.config/cursor)

Stores Cursor CLI configuration. Required only when using Cursor as the AI provider. This is a read-write mount as the Cursor CLI may write runtime state.

Health Checks

The container includes a built-in health check that probes the /health endpoint:

healthcheck:
  test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
  interval: 30s
  timeout: 10s
  retries: 3
Parameter Value Description
interval 30s Time between health check probes
timeout 10s Maximum wait time for a response
retries 3 Consecutive failures before marking unhealthy

The health check begins after the container starts. After 3 consecutive failures (90 seconds with default settings), Docker marks the container as unhealthy.

Checking Health Status

# View container health status
docker compose ps

# Inspect health check details
docker inspect --format='{{json .State.Health}}' docsfy-docsfy-1 | jq

Using Health Checks with Orchestrators

The health check integrates with orchestration tools:

  • Docker Compose: Use depends_on with condition: service_healthy to start dependent services only after docsfy is ready.
  • Kubernetes: Map the /health endpoint to a livenessProbe and readinessProbe.
  • Docker Swarm: Automatically restarts unhealthy containers.
# Example: dependent service waiting for docsfy
services:
  docsfy:
    build: .
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 30s
      timeout: 10s
      retries: 3

  nginx:
    image: nginx:alpine
    depends_on:
      docsfy:
        condition: service_healthy

Production Deployment Considerations

Persistent Storage

For production, use named Docker volumes instead of bind mounts for the data directory:

services:
  docsfy:
    build: .
    volumes:
      - docsfy-data:/data

volumes:
  docsfy-data:
    driver: local

Named volumes are managed by Docker and provide better performance on some platforms (notably Docker Desktop on macOS).

Reverse Proxy

In production, run docsfy behind a reverse proxy to handle TLS termination, rate limiting, and static asset caching:

upstream docsfy {
    server docsfy:8000;
}

server {
    listen 443 ssl;
    server_name docs.example.com;

    ssl_certificate     /etc/ssl/certs/cert.pem;
    ssl_certificate_key /etc/ssl/private/key.pem;

    location / {
        proxy_pass http://docsfy;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # Cache generated static documentation assets
    location /docs/ {
        proxy_pass http://docsfy;
        proxy_cache_valid 200 10m;
    }
}

Resource Limits

AI CLI operations are resource-intensive. Set appropriate container resource limits:

services:
  docsfy:
    build: .
    deploy:
      resources:
        limits:
          memory: 4G
          cpus: "2.0"
        reservations:
          memory: 1G
          cpus: "0.5"

Note: The AI_CLI_TIMEOUT environment variable (default: 60 minutes) controls how long individual AI CLI invocations can run. Adjust this based on the size of repositories you expect to process.

Security Hardening

  1. Read-only root filesystem: Mount the container filesystem as read-only, with explicit writable mounts for /data and /tmp:

    yaml services: docsfy: build: . read_only: true tmpfs: - /tmp volumes: - docsfy-data:/data

  2. Drop capabilities: Reduce the container's Linux capabilities to the minimum needed:

    yaml services: docsfy: cap_drop: - ALL cap_add: - NET_RAW # Required for health check curl

  3. Network isolation: Place docsfy in an internal network if it only needs to be reached via a reverse proxy:

    ```yaml services: docsfy: networks: - internal

    nginx: networks: - internal - external

    networks: internal: internal: true external: ```

Backup Strategy

The SQLite database and cached markdown files should be backed up regularly:

# Stop the service to ensure database consistency
docker compose stop docsfy

# Back up the data directory
tar czf docsfy-backup-$(date +%Y%m%d).tar.gz ./data/

# Restart the service
docker compose start docsfy

For zero-downtime backups, use SQLite's .backup command:

docker compose exec docsfy sqlite3 /data/docsfy.db ".backup '/data/docsfy-backup.db'"

Logging

Configure log output for production observability:

services:
  docsfy:
    build: .
    environment:
      - LOG_LEVEL=WARNING
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"

OpenShift Deployment

docsfy is designed to be OpenShift-compatible. The appuser runs with GID 0, which satisfies OpenShift's restricted SCC. No additional configuration is required — the image works under OpenShift's arbitrary UID assignment out of the box.
