Generating Documentation

Documentation generation in docsfy is triggered through a single API call. You provide a GitHub repository URL, and docsfy clones the repository, uses an AI CLI tool to analyze the codebase, and produces a polished static HTML documentation site.

This page covers how to start generation, monitor progress, and access the finished docs for both public and private repositories.

Quick Start

To generate documentation for a repository, send a POST request to the /api/generate endpoint with the repository URL:

curl -X POST http://localhost:8000/api/generate \
  -H "Content-Type: application/json" \
  -d '{"url": "https://github.com/your-org/your-repo"}'

docsfy will clone the repository and run through the full generation pipeline. You can monitor progress via the status endpoints and access the finished site once generation completes.

Generation Pipeline

Every generation request passes through four sequential stages:

Stage 1: Clone Repository

docsfy performs a shallow clone (--depth 1) of the repository to a temporary directory. This keeps clone times fast, even for large repositories with extensive commit histories.

Stage 2: AI Planner

The configured AI CLI runs with its working directory set to the cloned repo. It explores the entire repository and produces a plan.json file containing the documentation structure — pages, sections, and navigation hierarchy.

Stage 3: AI Content Generator

For each page defined in plan.json, the AI CLI runs again with access to the full codebase. Pages are generated concurrently using async execution with semaphore-limited concurrency. Each page is output as a Markdown file and cached at /data/projects/{name}/cache/pages/*.md.

Stage 4: HTML Renderer

Markdown pages and plan.json are converted into a polished static HTML site using Jinja2 templates with bundled CSS/JS assets. The final site includes sidebar navigation, dark/light theme toggle, client-side search, code syntax highlighting, and responsive design.

Public Repositories

Public repositories work out of the box with no additional configuration. Pass any public HTTPS URL:

curl -X POST http://localhost:8000/api/generate \
  -H "Content-Type: application/json" \
  -d '{"url": "https://github.com/fastapi/fastapi"}'

Both HTTPS and SSH URL formats are supported:

# HTTPS
{"url": "https://github.com/your-org/your-repo"}

# SSH
{"url": "git@github.com:your-org/your-repo.git"}

Private Repositories

Private repositories are supported through system git credentials. docsfy uses whatever git authentication is configured in the container environment.

SSH Key Authentication

Mount your SSH keys into the container so git can authenticate with GitHub over SSH:

services:
  docsfy:
    build: .
    ports:
      - "8000:8000"
    env_file: .env
    volumes:
      - ./data:/data
      - ~/.ssh:/home/appuser/.ssh:ro

Then use an SSH URL for the repository:

curl -X POST http://localhost:8000/api/generate \
  -H "Content-Type: application/json" \
  -d '{"url": "git@github.com:your-org/private-repo.git"}'

HTTPS with Stored Credentials

Alternatively, configure git credential storage in the container so HTTPS URLs can authenticate against private repositories. This can be done through git credential helpers or by mounting a .gitconfig with stored credentials.

Warning: When mounting credentials into the container, always use the :ro (read-only) flag to prevent accidental modification.

Monitoring Generation Status

List All Projects

Retrieve a list of all projects and their current generation status:

curl http://localhost:8000/api/status

Each project has one of three statuses:

Status Description
generating Pipeline is currently running
ready Documentation has been generated successfully
error Generation failed — check logs for details

Get Project Details

Retrieve details for a specific project, including last generated timestamp, commit SHA, and page list:

curl http://localhost:8000/api/projects/{name}

Tip: The project name is derived from the repository name. For example, https://github.com/your-org/my-app produces a project named my-app.

Accessing Generated Documentation

Serve Directly

Once generation completes, the documentation site is served directly from the docsfy API:

http://localhost:8000/docs/{project}/{path}

For example, to view the index page of a project called my-app:

http://localhost:8000/docs/my-app/index.html

Download as Archive

Download the entire generated site as a .tar.gz archive for self-hosting:

curl -O http://localhost:8000/api/projects/{name}/download

The archive contains a fully self-contained static HTML site that can be deployed to any static hosting provider (GitHub Pages, Netlify, S3, Nginx, etc.).

Regenerating Documentation

To regenerate documentation for a repository that has already been processed, call POST /api/generate again with the same URL. docsfy uses an incremental update strategy:

  1. Fetches the repository and compares the current commit SHA against the stored SHA
  2. If the repository has changed, re-runs the AI Planner to check for structural changes
  3. Regenerates only pages whose content may be affected by the changes
  4. If the plan structure is unchanged and only specific files changed, only the relevant pages are regenerated

This makes subsequent generation runs significantly faster than the initial run.

Note: If you want a full regeneration from scratch, delete the project first with DELETE /api/projects/{name} and then trigger a new generation.

Removing a Project

To remove a project and all its generated documentation:

curl -X DELETE http://localhost:8000/api/projects/{name}

This removes the project metadata from the database, deletes all cached Markdown, and removes the rendered HTML site.

AI Provider Configuration

docsfy supports three AI providers for documentation generation. Configure your provider using environment variables in your .env file.

Claude (Default)

AI_PROVIDER=claude
AI_MODEL=claude-opus-4-6[1m]

Authenticate with either an API key or Vertex AI:

# Option 1: API Key
ANTHROPIC_API_KEY=sk-ant-...

# Option 2: Google Cloud Vertex AI
CLAUDE_CODE_USE_VERTEX=1
CLOUD_ML_REGION=us-east5
ANTHROPIC_VERTEX_PROJECT_ID=your-gcp-project

When using Vertex AI, mount your Google Cloud credentials into the container:

volumes:
  - ~/.config/gcloud:/home/appuser/.config/gcloud:ro

Gemini

AI_PROVIDER=gemini
AI_MODEL=gemini-2.5-pro
GEMINI_API_KEY=your-api-key

Cursor

AI_PROVIDER=cursor
AI_MODEL=your-preferred-model
CURSOR_API_KEY=your-api-key

Mount the Cursor configuration directory:

volumes:
  - ./cursor:/home/appuser/.config/cursor

Timeout Configuration

All providers share a configurable timeout for AI CLI operations:

AI_CLI_TIMEOUT=60  # timeout in minutes (default: 60)

Note: Large repositories with many files may require a longer timeout. Increase AI_CLI_TIMEOUT if generation is timing out before completion.

Health Check

Verify that the docsfy service is running and available:

curl http://localhost:8000/health

This endpoint is also used by Docker's built-in health check mechanism as configured in docker-compose.yaml:

healthcheck:
  test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
  interval: 30s
  timeout: 10s
  retries: 3

API Reference Summary

Method Path Description
POST /api/generate Start documentation generation for a repository URL
GET /api/status List all projects and their generation status
GET /api/projects/{name} Get project details (last generated, commit SHA, pages)
DELETE /api/projects/{name} Remove a project and its generated docs
GET /api/projects/{name}/download Download the site as a .tar.gz archive
GET /docs/{project}/{path} Serve generated static HTML documentation
GET /health Health check

Storage Layout

Generated documentation is stored on disk under /data/projects/:

/data/projects/{project-name}/
  plan.json             # documentation structure from AI
  cache/
    pages/*.md          # AI-generated markdown (cached for incremental updates)
  site/                 # final rendered HTML
    index.html
    *.html
    assets/
      style.css
      search.js
      theme-toggle.js
      highlight.js
    search-index.json

Project metadata (status, commit SHA, timestamps, generation history) is stored in a SQLite database at /data/docsfy.db.

Tip: Mount /data as a persistent volume to preserve generated documentation across container restarts. This is already configured in the default docker-compose.yaml.
