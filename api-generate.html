POST /api/generate

POST /api/generate

Start documentation generation for a GitHub repository. This endpoint triggers the full generation pipeline — cloning the repository, planning the documentation structure with AI, generating page content, and rendering a polished static HTML site.

Endpoint

POST /api/generate

Request Body

Send a JSON object containing the GitHub repository URL to generate documentation for.

{
  "repo_url": "https://github.com/owner/repository"
}

Parameters

Field Type Required Description
repo_url string Yes The GitHub repository URL. Supports both HTTPS and SSH formats.

Supported URL Formats

Both public and private repositories are supported:

# HTTPS (public or private)
https://github.com/owner/repository

# SSH (private repos using system git credentials)
git@github.com:owner/repository.git

Note: Private repository access relies on system git credentials configured in the container environment. See Configuration for details on mounting credentials.

Response

A successful request returns 200 OK with the project metadata, including its current generation status.

{
  "name": "repository",
  "repo_url": "https://github.com/owner/repository",
  "status": "generating",
  "last_generated": null,
  "last_commit_sha": null
}

Response Fields

Field Type Description
name string The project name, derived from the repository name.
repo_url string The repository URL that was submitted.
status string Current generation status. One of generating, ready, or error.
last_generated string \| null ISO 8601 timestamp of the last successful generation, or null if not yet complete.
last_commit_sha string \| null The commit SHA used for the most recent generation, or null if not yet complete.

Status Values

Status Description
generating The documentation pipeline is currently running.
ready Documentation has been successfully generated and is available to view or download.
error Generation failed. Check project details for error logs.

Example

cURL

curl -X POST http://localhost:8000/api/generate \
  -H "Content-Type: application/json" \
  -d '{"repo_url": "https://github.com/myk-org/docsfy"}'

Python (requests)

import requests

response = requests.post(
    "http://localhost:8000/api/generate",
    json={"repo_url": "https://github.com/myk-org/docsfy"}
)

project = response.json()
print(f"Project '{project['name']}' status: {project['status']}")

Python (httpx async)

import httpx

async with httpx.AsyncClient() as client:
    response = await client.post(
        "http://localhost:8000/api/generate",
        json={"repo_url": "https://github.com/myk-org/docsfy"}
    )
    project = response.json()

Generation Pipeline

When a request is accepted, docsfy runs four sequential stages in the background:

Stage 1: Clone Repository

The repository is shallow-cloned (--depth 1) to a temporary directory. This minimizes disk usage and clone time while providing full access to the current source tree.

Stage 2: AI Planner

The configured AI CLI analyzes the entire repository and produces a plan.json file containing the documentation structure — pages, sections, and navigation hierarchy.

# The AI CLI runs with cwd set to the cloned repo directory
# Provider commands vary by configuration:

# Claude
claude --model <model> --dangerously-skip-permissions -p

# Gemini
gemini --model <model> --yolo

# Cursor
agent --force --model <model> --print --workspace <path>

Stage 3: AI Content Generator

For each page defined in plan.json, the AI CLI runs with full access to the cloned repository. Pages are generated concurrently using async execution with semaphore-limited concurrency. Generated markdown files are cached at /data/projects/{name}/cache/pages/*.md.

Stage 4: HTML Renderer

Markdown pages and plan.json are converted into a polished static HTML site using Jinja2 templates with bundled CSS/JS assets. The final site includes:

  • Sidebar navigation
  • Dark/light theme toggle
  • Client-side search (via lunr.js)
  • Code syntax highlighting (via highlight.js)
  • Callout boxes (note, warning, info)
  • Responsive design

Output is written to /data/projects/{name}/site/.

Tip: Once generation completes with status ready, view the docs at GET /docs/{project}/ or download them as a .tar.gz archive via GET /api/projects/{name}/download.

Incremental Updates

Calling POST /api/generate for a repository that has already been generated triggers an incremental update:

  1. The current commit SHA is compared against the stored SHA from the last generation.
  2. If the repository has changed, the AI Planner re-evaluates the documentation structure.
  3. Only pages affected by the changes are regenerated — unchanged pages use cached markdown.
  4. The HTML renderer rebuilds the full site with updated content.

Note: If the repository has not changed since the last generation, the endpoint returns the existing project metadata without re-running the pipeline.

Error Handling

HTTP Status Description
200 Generation started successfully (or project already exists).
422 Invalid request body — missing or malformed repo_url.
500 Internal server error during pipeline initialization.

If generation fails during any pipeline stage, the project status is set to error. Retrieve error details using GET /api/projects/{name}.

{
  "name": "repository",
  "repo_url": "https://github.com/owner/repository",
  "status": "error",
  "last_generated": null,
  "last_commit_sha": null
}

Warning: Generation is a long-running process that involves cloning a repository and making multiple AI CLI calls. The AI_CLI_TIMEOUT environment variable (default: 60 minutes) controls the maximum time allowed for each AI invocation.

Related Configuration

The AI provider and model used during generation are controlled by environment variables:

# AI Configuration
AI_PROVIDER=claude          # Options: claude, gemini, cursor
AI_MODEL=claude-opus-4-6[1m]  # Model identifier for the selected provider
AI_CLI_TIMEOUT=60           # Timeout per AI invocation in minutes

See the Configuration page for the full list of environment variables and provider setup instructions.
