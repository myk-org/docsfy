Contributing

Prerequisites

docsfy requires the following tools installed on your development machine:

  • Python 3.12+
  • uv — the sole package manager for this project (do not use pip)
  • Git

Warning: This project uses uv exclusively for dependency management. Do not use pip, pip-tools, or other package managers. All commands assume uv is available on your PATH.

Development Setup

1. Clone the Repository

git clone https://github.com/your-org/docsfy.git
cd docsfy

2. Create a Virtual Environment and Install Dependencies

uv venv
uv sync --dev

This installs all runtime and development dependencies defined in pyproject.toml, including linters, type checkers, and test frameworks.

3. Install Pre-commit Hooks

uv run pre-commit install

This registers the pre-commit hooks so they run automatically on every git commit. You can also run them manually against all files:

uv run pre-commit run --all-files

4. Set Up Environment Variables

Copy the example environment file and configure it for local development:

cp .env.example .env

At minimum, configure an AI provider for local testing:

# AI Configuration
AI_PROVIDER=claude
AI_MODEL=claude-opus-4-6[1m]
AI_CLI_TIMEOUT=60

# Claude - Option 1: API Key
# ANTHROPIC_API_KEY=

# Claude - Option 2: Vertex AI
# CLAUDE_CODE_USE_VERTEX=1
# CLOUD_ML_REGION=
# ANTHROPIC_VERTEX_PROJECT_ID=

# Gemini
# GEMINI_API_KEY=

# Cursor
# CURSOR_API_KEY=

# Logging
LOG_LEVEL=INFO

5. Run the Development Server

uv run uvicorn docsfy.main:app --host 0.0.0.0 --port 8000 --reload

The API will be available at http://localhost:8000 with auto-reload on code changes. The health check endpoint is at http://localhost:8000/health.

Build System

docsfy uses hatchling as its build backend. The project metadata and build configuration are defined in pyproject.toml:

[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

Tip: You can build the package locally with uv build to verify that packaging works correctly before submitting changes.

Pre-commit Hooks

Pre-commit hooks enforce code quality standards automatically before each commit. The project uses the following hooks:

Hook Purpose
ruff (lint) Fast Python linter — catches errors, style issues, and import sorting
ruff (format) Deterministic code formatter (replaces Black)
mypy (strict) Static type checking with strict mode enabled
flake8 Additional lint checks complementing ruff
gitleaks Scans for hardcoded secrets and credentials in commits
detect-secrets Prevents secrets from being committed to the repository
Standard hooks Trailing whitespace, end-of-file fixer, YAML/TOML validation, merge conflict detection

Ruff (Linter and Formatter)

Ruff serves as both the primary linter and code formatter. It replaces multiple tools (isort, Black, pyflakes, pycodestyle) with a single, fast Rust-based tool.

# Run the linter manually
uv run ruff check .

# Auto-fix linting issues where possible
uv run ruff check . --fix

# Run the formatter
uv run ruff format .

# Check formatting without applying changes
uv run ruff format . --check

Note: Ruff configuration is defined in pyproject.toml. Follow the existing rule selections and per-file ignores when adding new modules.

mypy (Strict Type Checking)

mypy runs in strict mode, which means all functions must have type annotations and no implicit Any types are allowed.

uv run mypy .

When writing new code, ensure all function signatures include type annotations:

# Good — fully annotated
async def generate_docs(repo_url: str, project_name: str) -> tuple[bool, str]:
    ...

# Bad — missing annotations (mypy strict will reject this)
async def generate_docs(repo_url, project_name):
    ...

Tip: If you must use a third-party library that lacks type stubs, add the appropriate # type: ignore[import-untyped] comment rather than disabling mypy for the entire file.

Flake8

Flake8 provides additional lint checks that complement ruff's rule set:

uv run flake8 .

Gitleaks and detect-secrets

These hooks scan your commits for accidentally included secrets such as API keys, tokens, and passwords.

# Run gitleaks manually
uv run pre-commit run gitleaks --all-files

# Run detect-secrets
uv run pre-commit run detect-secrets --all-files

Warning: Never commit API keys, credentials, or secrets to the repository. Use the .env file for local configuration — it is excluded via .gitignore. If gitleaks or detect-secrets blocks your commit, review the flagged content and either remove the secret or add a legitimate false positive to the allowlist.

Running All Hooks

To run the full pre-commit suite against every file (not just staged changes):

uv run pre-commit run --all-files

Tox Test Runner

Tox orchestrates test environments and quality checks. It is configured to use uv as the underlying package manager.

Available Tox Environments

Environment Purpose
Unit tests Runs the full test suite via pytest
Unused code Detects dead code that can be safely removed

Running Tests

# Run all tox environments
uv run tox

# Run only unit tests
uv run tox -e py

# Run only the unused-code check
uv run tox -e unused

# Run with verbose output
uv run tox -e py -- -v

Writing Tests

Tests live in the tests/ directory and use pytest. Follow these conventions:

  • Mirror the src/ directory structure under tests/ (e.g., src/docsfy/ai.py → tests/test_ai.py)
  • Use descriptive test function names: test_<function>_<scenario>_<expected>
  • Use fixtures for shared setup
  • Use pytest.mark.asyncio for async test functions

Example:

import pytest
from docsfy.ai import invoke_ai_cli


@pytest.mark.asyncio
async def test_invoke_ai_cli_returns_success_on_valid_prompt() -> None:
    success, output = await invoke_ai_cli(prompt="Say hello")
    assert success is True
    assert len(output) > 0


def test_parse_json_response_handles_markdown_code_blocks() -> None:
    raw = '```json\n{"pages": []}\n```'
    result = parse_json_response(raw)
    assert result == {"pages": []}

Note: The project uses asyncio.to_thread(subprocess.run, ...) for AI CLI invocation. When testing these paths, mock subprocess.run to avoid actual AI CLI calls during tests.

Code Quality Standards

Python Version

All code must be compatible with Python 3.12+. Use modern Python features:

  • Type hints (required — mypy strict is enforced)
  • match statements where appropriate
  • f-strings for string formatting
  • dataclass(frozen=True) for immutable configuration objects
  • async/await for I/O-bound operations

Project Structure

src/docsfy/
  main.py              # FastAPI application and endpoint definitions
  ai.py                # AI CLI provider integration
  pipeline.py          # Generation pipeline orchestration
  renderer.py          # Markdown-to-HTML conversion
  models.py            # Pydantic models and database schemas
  templates/           # Jinja2 HTML templates
  static/              # CSS, JS assets

Configuration Objects

Use frozen dataclasses for provider configurations, following the established pattern:

@dataclass(frozen=True)
class ProviderConfig:
    binary: str
    build_cmd: Callable
    uses_own_cwd: bool = False

Import Ordering

Ruff handles import sorting automatically. Imports follow the standard convention:

  1. Standard library imports
  2. Third-party imports
  3. Local application imports

Each group is separated by a blank line.

Error Handling

  • Use specific exception types, not bare except:
  • AI CLI invocations return tuple[bool, str] (success flag, output) — do not raise exceptions for expected failures
  • Log errors with appropriate severity levels using Python's logging module

Docker Development

For running the full stack locally with Docker:

docker compose up --build

The docker-compose.yaml mounts the necessary volumes:

services:
  docsfy:
    build: .
    ports:
      - "8000:8000"
    env_file: .env
    volumes:
      - ./data:/data
      - ~/.config/gcloud:/home/appuser/.config/gcloud:ro
      - ./cursor:/home/appuser/.config/cursor
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 30s
      timeout: 10s
      retries: 3

The container runs as a non-root user (appuser) and is OpenShift-compatible (GID 0).

How to Submit Changes

1. Create a Feature Branch

git checkout -b feature/your-feature-name

Use a descriptive branch name with one of these prefixes:

Prefix Use for
feature/ New features and enhancements
fix/ Bug fixes
refactor/ Code restructuring without behavior changes
docs/ Documentation updates
chore/ Build, CI, and tooling changes

2. Make Your Changes

  • Write code following the quality standards described above
  • Add or update tests for any changed behavior
  • Ensure all type annotations are present (mypy strict)

3. Verify Locally

Before pushing, run the full quality suite:

# Run all pre-commit hooks
uv run pre-commit run --all-files

# Run the full tox suite
uv run tox

Both commands must pass cleanly.

4. Commit Your Changes

Write clear, descriptive commit messages:

git add <files>
git commit -m "feat: add incremental update support for changed pages"

Note: Pre-commit hooks run automatically on git commit. If a hook fails, fix the issue and re-commit. Do not bypass hooks with --no-verify.

5. Push and Open a Pull Request

git push -u origin feature/your-feature-name

Open a pull request against the main branch. In your PR description:

  • Summarize what changed and why
  • Note any new dependencies added
  • Call out architectural decisions or trade-offs
  • Reference related issues if applicable

6. Code Review

All pull requests require review before merging. Reviewers will check:

  • Tests pass and cover new/changed behavior
  • Type annotations are complete (mypy strict passes)
  • No secrets or credentials are included
  • Code follows established patterns in the codebase
  • Documentation is updated if user-facing behavior changes
